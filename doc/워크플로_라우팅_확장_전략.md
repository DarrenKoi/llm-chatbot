# 워크플로 수가 많아질 때 라우팅/제어 전략 연구

## 1. 왜 지금 구조가 버거워지는가

현재 저장소의 루트 진입점은 `api/workflows/start_chat/lg_graph.py`의 `classify_node()`이고, 이 함수는 각 워크플로의 `handoff_keywords`를 순회하면서 사용자 메시지에 부분 문자열이 포함되는지 확인해 `active_workflow`를 정한다.

이 방식은 워크플로가 몇 개 없을 때는 빠르고 설명 가능하다. 하지만 워크플로 수가 늘어나면 아래 문제가 빠르게 커진다.

- 같은 의미를 여러 표현으로 말하는 자연어 특성 때문에 키워드 목록이 계속 늘어난다.
- 서로 다른 워크플로가 비슷한 키워드를 공유하기 시작한다.
- 한국어/영어 혼합, 띄어쓰기 변형, 구어체, 오탈자 대응이 약하다.
- 현재 대화 맥락 없이 현재 발화만 보고 라우팅하므로 후속 질문 처리 품질이 떨어진다.
- 모든 handoff 워크플로를 루트 그래프에 붙여 컴파일하는 구조라, 워크플로 수가 많아질수록 루트 그래프의 비대화가 발생한다.

정리하면, "키워드 매칭 자체"가 문제라기보다 "키워드 매칭만으로 전체 라우팅을 책임지게 하는 구조"가 문제다.

## 2. 외부 자료에서 확인되는 공통 패턴

여러 제품 문서는 공통적으로 아래 원칙을 보여준다.

- 단순 규칙만으로 끝내지 않는다.
- intent 분류에는 confidence threshold, no-match, negative examples 같은 안전장치를 둔다.
- 관련 intent/workflow를 계층적으로 묶고, 현재 컨텍스트에 따라 후보를 줄인다.
- 워크플로 자체도 하나의 거대한 루트가 아니라 재사용 가능한 서브플로/서브그래프로 분해한다.
- 운영 단계에서는 no-match, 오분류, 애매한 분기 데이터를 계속 수집해서 taxonomy와 예시 데이터를 다듬는다.

이 문서의 추천안은 위 원칙을 현재 저장소 구조에 맞춰 옮긴 것이다.

## 3. 전략별 비교

### 전략 A. 규칙 기반 라우터를 더 엄격하게 운영한다

가장 보수적인 확장 방법이다. 단순 키워드 포함 여부에서 끝내지 않고, 규칙을 명시적인 라우팅 정책으로 승격한다.

예시:

- 정확 명령어 매칭
- 정규식 매칭
- 우선순위(priority)
- 금지 키워드(negative keywords)
- 현재 활성 컨텍스트가 있을 때만 허용되는 규칙
- 사용자 권한/채널/언어별 allowlist

권장 메타데이터:

- `workflow_id`
- `domain_id`
- `handoff_keywords`
- `handoff_patterns`
- `negative_examples`
- `priority`
- `supported_locales`
- `requires_active_context`
- `allow_global_entry`

장점:

- 설명 가능성이 높다.
- 비용이 낮고 지연이 작다.
- 운영자가 오동작 원인을 추적하기 쉽다.
- 민감한 워크플로에 안전장치를 두기 좋다.

단점:

- 워크플로 수가 늘수록 규칙 충돌 관리 비용이 커진다.
- 표현 다양성에 약하다.
- 유지보수 지식이 특정 사람 머릿속에 남기 쉽다.
- 새로운 워크플로 추가 시 규칙 회귀가 자주 생긴다.

언제 적합한가:

- 운영 리스크가 큰 명령형 워크플로
- 명확한 trigger phrase가 있는 워크플로
- 초기에 빠르게 품질을 안정화해야 할 때

한 줄 평가:

`keyword matching`의 "개선판"으로는 유효하지만, 워크플로가 많아지는 문제의 최종 해법은 아니다.

### 전략 B. intent classifier를 도입한다

각 워크플로를 하나의 intent처럼 보고, 예시 문장(training phrases)을 모아 분류 모델 또는 LLM classifier가 가장 적절한 워크플로를 선택하게 하는 방식이다.

이 접근은 Dialogflow CX가 쓰는 기본 모델과 가장 가깝다. Dialogflow CX 문서는 training phrases를 기반으로 유사 발화를 일반화해 intent를 매칭하고, confidence threshold를 넘지 못하면 `no-match`로 처리한다고 설명한다. 또한 negative intent를 통해 "비슷해 보이지만 어떤 intent에도 매칭되면 안 되는 문장"을 따로 관리하도록 권한다.

장점:

- 표현이 다양해져도 키워드 방식보다 일반화가 잘 된다.
- 한국어/영어 혼합, 오탈자, 간접 표현 대응이 좋아질 수 있다.
- 새 워크플로 추가 시 규칙 몇 개를 늘리는 대신 예시 데이터를 추가하는 방식으로 확장할 수 있다.

단점:

- 예시 문장 품질과 분포에 따라 성능이 흔들린다.
- confidence threshold, ambiguity threshold, fallback 정책이 없으면 오분류가 커진다.
- 데이터셋 관리와 정기 평가 체계가 필요하다.
- 워크플로 수가 많아질수록 비슷한 intent 간 경계 설계가 중요해진다.

언제 적합한가:

- 표현 다양성이 크고 키워드가 자주 빗나가는 경우
- 워크플로 정의를 intent taxonomy로 정리할 수 있는 경우

한 줄 평가:

규칙 기반보다 확장성이 좋지만, "분류기만 붙이면 끝"은 아니다. 반드시 fallback과 오분류 관측 체계가 같이 와야 한다.

### 전략 C. 계층형 라우팅을 쓴다

처음부터 모든 워크플로를 N-way 분류하지 않고, 먼저 큰 도메인을 고르고 그 다음에 해당 도메인 내부에서 세부 워크플로를 고르는 방식이다.

예시:

1. 전역 라우터: `language`, `travel`, `analytics`, `general_chat`
2. 도메인 라우터: `language` 내부에서 `translator`, `proofreader`, `summarizer`
3. 워크플로 내부 resolve: 필요한 슬롯 수집 후 실제 실행

Dialogflow CX는 head intent와 supplemental intent를 구분하고, route group과 scope 안에서 intent를 평가한다. Dialogflow ES의 contexts 개념도 "현재 맥락이 활성화된 상태에서만 특정 intent를 더 잘 매칭"하도록 만든다. 이 철학은 워크플로 라우팅에도 그대로 적용할 수 있다.

장점:

- 후보 수를 단계적으로 줄여서 충돌을 줄인다.
- 도메인별 팀 소유권을 분리하기 쉽다.
- 문서화와 운영이 좋아진다.
- 후속 질문 처리에 강하다.

단점:

- 라우터가 두 단계 이상이 되면서 구조가 복잡해진다.
- 상위 단계에서 잘못 분기하면 하위 단계가 아무리 좋아도 복구가 어렵다.
- 도메인 taxonomy 설계가 부실하면 오히려 혼란이 커진다.

언제 적합한가:

- 워크플로가 10개를 넘기 시작하고, 기능군이 자연스럽게 나뉘는 경우
- follow-up 질문과 맥락 기반 분기가 많은 경우

한 줄 평가:

워크플로가 많아질수록 가장 먼저 고려해야 하는 구조적 개선이다.

### 전략 D. semantic retrieval 기반 후보 선택을 쓴다

각 워크플로를 "설명 가능한 객체"로 보고, 설명/예시/지원 범위/필수 슬롯/금지 조건 같은 메타데이터를 문서처럼 저장한 뒤, 사용자 발화와 가장 가까운 워크플로 후보를 top-k로 검색하는 방식이다.

예시 워크플로 문서:

- workflow name
- description
- good examples
- bad examples
- required inputs
- output type
- supported locales
- safety notes

LlamaIndex의 RouterQueryEngine 계열 문서는 selector가 각 후보의 metadata와 query를 바탕으로 후보를 고른다고 설명한다. 즉, "코드에 박힌 keyword list" 대신 "검색 가능한 workflow catalog"를 두는 접근이다.

장점:

- 새 워크플로 추가가 비교적 쉽다.
- 긴 설명이나 복합 목적도 후보 검색에 반영할 수 있다.
- exact keyword보다 의미 기반으로 후보를 좁히기 좋다.
- top-k 후보를 남겨 후단 검증 단계와 결합하기 쉽다.

단점:

- retrieval 품질이 불안정하면 엉뚱한 후보가 top-k에 들어간다.
- metadata 작성 품질이 낮으면 효과가 떨어진다.
- 단일 정답보다 "후보 생성" 단계로 쓰는 것이 적합하다.
- 평가 체계 없이 쓰면 왜 특정 후보가 뽑혔는지 설명이 어려울 수 있다.

언제 적합한가:

- 워크플로 수가 많고 이름만으로는 구분이 어려운 경우
- 각 워크플로를 설명/예시 중심 catalog로 관리하고 싶은 경우

한 줄 평가:

단독 최종 라우터보다 `candidate generation` 용도로 매우 유용하다.

### 전략 E. LLM judge 또는 selector를 최종 판정기로 쓴다

top-k 후보가 정리된 뒤, LLM에게 각 후보 설명과 사용자 발화를 보여주고 어떤 워크플로가 맞는지 선택하게 하는 방식이다.

보통 아래처럼 쓴다.

1. 규칙 또는 retrieval로 후보 3~5개 생성
2. LLM selector가 후보 중 하나를 고름
3. score가 낮거나 이유가 빈약하면 clarification 또는 fallback

장점:

- 복합 의도나 장문의 요청 처리에 강하다.
- metadata를 잘 주면 자연스러운 설명 가능한 선택 이유를 만들 수 있다.
- 분류기 재학습 없이도 후보 설명 갱신만으로 품질을 높일 수 있다.

단점:

- 비용과 지연이 가장 크다.
- 출력 안정성 관리가 필요하다.
- 후보가 너무 많으면 prompt가 커지고 선택 품질이 떨어진다.
- 안전장치 없이 전면 적용하면 디버깅이 어렵다.

언제 적합한가:

- 규칙과 classifier만으로는 충돌이 잦은 중간 난이도 영역
- top-k 후보를 좁힌 뒤 최종 판단이 필요한 경우

한 줄 평가:

전체 라우팅을 LLM 하나에 맡기기보다, "후보가 줄어든 뒤 최종 선택"에 쓰는 편이 현실적이다.

### 전략 F. fallback과 clarification을 1급 기능으로 둔다

많은 시스템이 놓치는 부분인데, 워크플로가 많아질수록 "바로 분기하지 않는 능력"이 중요하다.

Rasa는 낮은 confidence, 또는 top-2 intent 간 점수 차가 너무 작은 경우 fallback을 발생시키는 정책을 문서화하고 있다. 또한 two-stage fallback처럼 "이 의도가 맞는지 확인 -> 아니면 재진술 요청 -> 그래도 불확실하면 최종 fallback" 흐름도 제공한다.

장점:

- 잘못된 워크플로 진입을 줄일 수 있다.
- 데이터 수집 품질이 좋아진다.
- 사용자가 의도를 직접 좁혀 주기 때문에 장기적으로 taxonomy 개선에 도움이 된다.

단점:

- 한 턴이 더 필요해 UX가 길어진다.
- 너무 자주 뜨면 시스템이 멍청해 보인다.
- 임계값을 잘못 잡으면 과도한 질문 또는 과도한 오분류가 생긴다.

언제 적합한가:

- 비슷한 워크플로가 많은 경우
- 잘못 진입했을 때 비용이 큰 경우

한 줄 평가:

라우터가 좋아질수록 clarification이 덜 필요해지는 것이 아니라, 더 정교한 clarification 정책이 필요해진다.

### 전략 G. 워크플로 실행 구조도 모듈화한다

라우팅 품질과 별개로, 워크플로 개수가 많아지면 오케스트레이션 구조 자체도 분해해야 한다.

LangGraph 문서는 subgraph를 쓰는 이유로 재사용, 독립 개발, 서로 다른 state schema 간 래핑을 명시한다. 또한 subgraph persistence를 `per-invocation`, `per-thread`, `stateless`로 나누어 설명한다. AWS Step Functions도 nested workflows를 통해 상위/하위 워크플로를 분리하고, 반복 요소를 재사용하며, 더 작은 state machine으로 복잡도를 줄이라고 권한다.

장점:

- 루트 그래프의 복잡도를 줄일 수 있다.
- 도메인별 배포와 디버깅이 쉬워진다.
- 재사용 가능한 workflow library를 만들 수 있다.
- 상태 스키마 경계를 명확히 하기 좋다.

단점:

- graph 경계를 넘는 데이터 계약을 잘 설계해야 한다.
- trace/로그를 한 눈에 묶는 작업이 필요하다.
- 너무 잘게 쪼개면 오히려 전체 흐름이 보기 어려워질 수 있다.

언제 적합한가:

- 도메인별 워크플로 수가 늘고 서로 다른 팀이 관리하기 시작할 때
- 루트 그래프 compile 비용과 디버깅 난도가 커질 때

한 줄 평가:

워크플로 수가 늘어날수록 라우터만 고치는 것으로는 부족하고, 실행 구조도 함께 분해해야 한다.

## 4. 무엇이 "proper way"인가

단일 정답은 없지만, 운영 경험이 쌓인 시스템들이 대체로 수렴하는 방향은 있다.

### 결론

워크플로가 많아질 때의 proper way는 아래 조합이다.

1. 전역 단일 키워드 매칭을 버리고 `계층형 라우팅`으로 바꾼다.
2. 라우팅은 `규칙 + classifier/retrieval + fallback`의 혼합형으로 만든다.
3. 워크플로 정의를 코드 내부 상수만이 아니라 `catalog/registry metadata`로 관리한다.
4. 루트 그래프는 얇게 유지하고, 실제 업무 실행은 `도메인 루트 + 하위 서브그래프`로 분리한다.
5. low-confidence와 ambiguous case는 `clarification`으로 처리한다.
6. no-match, wrong-handoff, clarification 데이터를 계속 수집해 taxonomy와 examples를 개선한다.

즉, 정답은 "키워드를 더 많이 넣는 것"이 아니라 "라우팅을 다단계 의사결정 시스템으로 바꾸는 것"이다.

## 5. 이 저장소에 맞는 권장 구조

### 5.1 권장 라우팅 파이프라인

현재 저장소 기준으로는 아래 구조가 가장 현실적이다.

1. `resume/interrupt` 우선
2. 명시 규칙 우선
3. 도메인 후보 선택
4. 워크플로 후보 선택
5. low-confidence면 clarification
6. 최종 handoff

세부 설명:

1. `resume/interrupt` 우선
   현재처럼 이미 진행 중인 워크플로가 있고 재개 가능한 상태면 새 라우팅보다 resume이 우선이다.
2. 명시 규칙 우선
   `cancel`, `stop`, 운영 명령, 관리자 명령, 명확한 slash-command형 요청은 규칙 기반으로 바로 처리한다.
3. 도메인 후보 선택
   `language`, `travel`, `analytics`, `general_chat` 같은 coarse domain을 먼저 정한다.
4. 워크플로 후보 선택
   도메인 내부에서 classifier 또는 semantic retrieval로 top-k 후보를 고른다.
5. low-confidence면 clarification
   상위 2개 후보가 비슷하거나 threshold 미달이면 사용자에게 한 번 확인한다.
6. 최종 handoff
   확정된 workflow만 실제 서브그래프로 보낸다.

### 5.2 권장 registry schema

`api/workflows/registry.py`는 이미 동적 발견 구조가 있으므로, 여기에 아래 메타데이터를 더하는 방향이 좋다.

- `workflow_id`
- `domain_id`
- `display_name`
- `description`
- `handoff_keywords`
- `handoff_patterns`
- `positive_examples`
- `negative_examples`
- `required_slots`
- `supported_locales`
- `priority`
- `allow_global_entry`
- `resume_policy`
- `clarification_hint`

핵심은 `handoff_keywords`만으로 판단하지 말고, 라우팅에 필요한 설명과 운영 정책을 함께 저장하는 것이다.

### 5.3 권장 그래프 구조

현재:

- `start_chat` 루트 그래프
- 모든 handoff workflow를 직접 서브그래프로 연결

권장:

- `start_chat`는 얇은 전역 라우터 역할만 수행
- 도메인 루트 그래프를 둠
  - 예: `language_root`, `travel_root`, `analytics_root`
- 각 도메인 아래에서 세부 workflow로 분기
- 무거운 워크플로는 lazy compile 또는 별도 실행 경계 고려

이렇게 하면 라우팅 정책 변경과 업무 워크플로 변경이 덜 얽힌다.

### 5.4 상태 관리 원칙

LangGraph 문서 기준으로 보면 현재처럼 멀티턴이 필요한 워크플로는 `per-thread` 성격의 기억이 필요할 수 있고, 단발성 워크플로는 `per-invocation`이 더 안전하다.

권장 원칙:

- 루트 공통 상태는 최소화
- workflow 고유 상태는 각 워크플로 내부에 둠
- 모든 워크플로가 같은 거대한 상태 dict를 공유하지 않음
- 멀티턴이 필요한 워크플로만 thread 단위 기억을 유지
- 독립 호출 성격이 강한 워크플로는 호출 단위 격리를 유지

## 6. 단계별 현실적 도입 순서

### 1단계. 키워드 라우터를 "정책 라우터"로 승격

지금 바로 할 수 있는 최소 개선이다.

- `handoff_keywords` 외에 `priority`, `negative_examples`, `domain_id` 추가
- `classify_node()`를 단순 substring 루프에서 score 기반 룰 엔진으로 교체
- wrong-handoff와 no-match 로그를 수집
- ambiguous case를 그냥 `start_chat`로 흘리지 말고 clarification 후보로 기록

장점:

- 구현 비용이 낮다.
- 현재 구조를 거의 유지할 수 있다.

한계:

- 표현 다양성 문제는 여전히 남는다.

### 2단계. domain classifier + workflow catalog 추가

실질적인 확장 구간이다.

- domain 분류 추가
- workflow metadata catalog 구축
- positive/negative example 정비
- semantic retrieval 또는 lightweight classifier로 top-k 후보 생성
- clarification flow 추가

장점:

- 워크플로 수가 늘어나도 운영 가능성이 크게 좋아진다.

한계:

- 데이터와 평가 체계가 필요하다.

### 3단계. hybrid router 완성

- 규칙 fast-path
- semantic top-k
- selector/LLM judge
- confidence threshold + ambiguity threshold
- clarification 또는 fallback

장점:

- 정확도와 운영성을 균형 있게 가져갈 수 있다.

한계:

- 구현 요소가 많아진다.

### 4단계. 도메인별 루트 그래프와 lazy execution 분리

- `start_chat`를 얇게 유지
- 도메인 루트 그래프 도입
- 큰 워크플로를 분리 배치
- tracing/observability 정비

장점:

- 워크플로 수가 많아져도 아키텍처가 덜 무너진다.

한계:

- 실행 경계와 상태 계약을 설계해야 한다.

## 7. 추천안

이 저장소에는 아래 조합을 추천한다.

### 추천 1. 단기

지금 당장은 키워드 매칭을 완전히 없애지 말고, 아래로 바꾸는 것이 좋다.

- `handoff_keywords` -> `routing_policy metadata`
- 전역 N-way 분기 -> `domain -> workflow` 2단계 분기
- 묵시적 실패 -> `clarification / no-match` 명시화

이 단계에서는 규칙 기반을 유지하되, 더 이상 "문자열 포함 여부" 하나에 기대지 않는 것이 핵심이다.

### 추천 2. 중기

워크플로 catalog를 만들고 semantic top-k 후보 생성을 붙이는 것이 좋다.

이유:

- 새 워크플로를 추가할 때 코드보다 메타데이터와 예시 중심으로 관리할 수 있다.
- 비슷한 표현이 많은 한국어 대화 입력에서 keyword보다 견고하다.
- 이후 LLM selector를 붙이기도 쉬워진다.

### 추천 3. 장기

루트 그래프를 얇게 만들고 domain root 아래로 서브그래프를 정리하는 것이 좋다.

이유:

- 라우팅 문제와 실행 문제를 분리할 수 있다.
- workflow 수가 많아질수록 compile/debug 부담을 줄일 수 있다.
- 팀별 ownership을 나누기 쉬워진다.

## 8. 추천하지 않는 방향

아래 방식은 피하는 편이 좋다.

- 모든 워크플로를 계속 같은 루트 그래프에 직접 연결하기
- 키워드 사전만 계속 키우기
- confidence나 ambiguity 개념 없이 classifier를 바로 production에 넣기
- fallback 없이 무조건 하나의 workflow를 강제 선택하기
- metadata 없이 LLM에게 전체 workflow 목록을 매번 던져 선택시키기

이 방식들은 초반에는 빨라 보여도, 워크플로가 많아지면 운영 비용이 폭발한다.

## 9. 실무 체크리스트

워크플로를 새로 추가할 때 아래를 같이 보아야 한다.

- 이 워크플로는 어느 `domain_id`에 속하는가
- 전역 진입이 가능한가, 아니면 특정 컨텍스트에서만 가능한가
- 대표 예시 문장 10개 이상이 준비되었는가
- 헷갈리기 쉬운 negative examples가 있는가
- low-confidence일 때 어떤 clarification 문구를 보여줄 것인가
- resume 가능한 멀티턴 workflow인가
- 기존 workflow와 충돌하는 trigger가 있는가
- no-match/오분류를 어떻게 관측할 것인가

## 10. 최종 요약

워크플로가 많아질 때 proper way는 "더 많은 키워드"가 아니라 "계층형 + 혼합형 + 모듈형"이다.

- 계층형: domain -> workflow -> slot-filling
- 혼합형: rule + classifier/retrieval + fallback
- 모듈형: 얇은 루트 라우터 + 도메인 루트 + 서브그래프

현재 저장소는 이미 동적 workflow registry와 LangGraph 기반 서브그래프 구조를 갖고 있으므로, 기반은 나쁘지 않다. 다음 단계는 keyword matching을 없애는 것이 아니라, 그것을 전체 라우팅 시스템의 한 층으로 격하시켜 더 큰 라우팅 정책 안에 넣는 것이다.
