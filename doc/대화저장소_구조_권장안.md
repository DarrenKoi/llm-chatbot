# 대화 저장소 구조 권장안

## 1. 목적

현재 프로젝트는 `api/conversation_service.py`에서 Redis를 이용해 사용자별 대화 history를 저장하고 있다.  
앞으로 사용할 수 있는 저장소가 `Redis(1.5GB Memory)`, `MongoDB(10GB)`, `OpenSearch(100GB)`라면, 대화 이력 시스템과 향후 RAG 지식검색 시스템은 아래 원칙으로 나누는 것이 가장 안정적이다.

- `MongoDB`: 대화 원본 저장소, 정합성 기준
- `OpenSearch`: 검색 전용 인덱스, RAG 검색 계층
- `Redis`: 캐시, 큐, 락, 단기 상태 저장

이 문서는 위 3개 저장소를 어떻게 역할 분리해서 운영할지, 그리고 현재 코드 기준으로 어떤 식으로 옮겨가면 되는지를 설명한다.

---

## 2. 결론

### 권장 역할 분리

| 저장소 | 권장 역할 | 비권장 역할 |
|------|------|------|
| MongoDB | 대화 원문 저장, 세션 메타데이터 저장, 보존 정책 관리 | 초고속 검색 엔진 역할 |
| OpenSearch | 전문 검색, 조건 검색, 운영 집계, 검색 랭킹, RAG retrieval | 원본 데이터 저장소 |
| Redis | 최근 대화 캐시, CDN 메타데이터 캐시, 큐, 분산 락, rate limit | 장기 대화 보관 |

### 한 줄 요약

- **대화는 MongoDB에 저장**
- **검색과 RAG retrieval은 OpenSearch에 위임**
- **Redis는 성능 최적화용 보조 계층으로만 사용**

---

## 3. 왜 이 구조가 맞는가

### 3.1 MongoDB가 메인 DB로 적합한 이유

대화 이력은 보통 아래 특성을 가진다.

- 메시지가 시간 순서대로 계속 append 된다.
- 메시지마다 메타데이터가 조금씩 다를 수 있다.
- 사용자별, 세션별, 기간별 조회가 많다.
- 보존 기간, TTL, 삭제 정책이 필요하다.
- 원문을 잃으면 안 되므로 정합성이 중요하다.

이 특성에는 문서 기반 구조가 잘 맞는다. MongoDB는 다음 장점이 있다.

- 스키마 변경에 유연하다.
- 세션 문서 + 메시지 문서 구조로 모델링하기 쉽다.
- `user_id`, `conversation_id`, `created_at` 기준 조회가 자연스럽다.
- TTL 인덱스를 붙여 보존 정책을 구현할 수 있다.
- OpenSearch보다 원본 저장소로 운영하기 쉽다.

### 3.2 OpenSearch가 메인 DB가 아니어야 하는 이유

OpenSearch는 검색에는 강하고, RAG 검색 계층으로도 유용하지만, 원본 저장소로는 불리하다.

- 검색 최적화 저장 구조라서 업데이트/부분 수정 모델이 복잡해진다.
- 최종 정합성 기준점으로 삼기 어렵다.
- 색인 실패나 재색인 이슈를 원본 저장 문제로 확대시키기 쉽다.
- 저장 용량이 커 보여도 검색 인덱스는 원문보다 디스크를 더 많이 먹는 경우가 많다.

따라서 OpenSearch는 "원본의 사본"이어야 하고, 원본 기준은 MongoDB에 둔다.

### 3.3 Redis는 왜 캐시/상태 저장에만 쓰는가

Redis는 매우 빠르지만 메모리 자원은 가장 작고 비싸다.

- 보유 자원: `1.5GB`
- 대화 원문은 누적될수록 계속 커진다.
- 현재 코드처럼 사용자별 history 전체를 Redis에 저장하면, 사용량이 늘수록 메모리 압박이 커진다.

따라서 Redis는 다음 용도가 더 적합하다.

- 최근 대화 N개 캐시
- CDN 메타데이터 캐시
- 큐/작업 상태
- 분산 락
- rate limit / idempotency key / dedup

## 4. 목표 아키텍처

```text
Cube / API Client
    |
    v
Flask App
    |
    +-- 1) MongoDB 저장
    |       - conversation_sessions
    |       - conversation_messages
    |
    +-- 2) Redis 캐시 갱신
    |       - recent history
    |       - CDN metadata
    |       - queue / lock / dedup
    |
    +-- 3) OpenSearch 비동기 인덱싱
            - message_search
            - conversation_summary
            - knowledge_chunks
            - knowledge_documents
```

### 처리 원칙

1. 요청을 받으면 먼저 MongoDB에 저장한다.
2. MongoDB 저장이 성공하면 Redis 캐시를 갱신한다.
3. OpenSearch 인덱싱은 비동기로 처리한다.
4. 검색 결과의 상세 원문이 필요하면 MongoDB를 다시 조회한다.

이 순서를 유지하면 저장과 검색이 분리되고, 검색 지연이 원본 저장 실패로 번지지 않는다.

---

## 5. MongoDB 설계

MongoDB는 `세션 컬렉션`과 `메시지 컬렉션`을 분리하는 편이 운영상 가장 안전하다.

### 5.1 권장 컬렉션

#### `conversation_sessions`

대화방 또는 대화 세션의 메타데이터를 저장한다.

예시 필드:

```json
{
  "_id": "conv_20260320_001",
  "user_id": "u12345",
  "channel": "cube",
  "title": "급여 공제 문의",
  "status": "open",
  "message_count": 14,
  "last_message_at": "2026-03-20T08:12:33Z",
  "tags": ["payroll", "hr"],
  "summary": "사용자가 급여 공제 내역과 계산 기준을 문의함",
  "created_at": "2026-03-20T08:10:01Z",
  "updated_at": "2026-03-20T08:12:33Z",
  "expires_at": "2026-09-20T00:00:00Z"
}
```

권장 인덱스:

- `{ user_id: 1, last_message_at: -1 }`
- `{ status: 1, last_message_at: -1 }`
- `{ expires_at: 1 }` with TTL if 보존 기간 자동 삭제 필요

#### `conversation_messages`

실제 메시지 원문을 한 건씩 저장한다.

예시 필드:

```json
{
  "_id": "msg_20260320_000001",
  "conversation_id": "conv_20260320_001",
  "user_id": "u12345",
  "channel": "cube",
  "message_id": "cube-msg-777",
  "role": "user",
  "content": "지난달 급여에서 공제된 항목 설명해줘",
  "tokens": null,
  "attachments": [],
  "metadata": {
    "cube_room_id": "room-1",
    "model": null,
    "trace_id": "req-abc"
  },
  "created_at": "2026-03-20T08:10:03Z",
  "expires_at": "2026-09-20T00:00:00Z"
}
```

권장 인덱스:

- `{ conversation_id: 1, created_at: 1 }`
- `{ user_id: 1, created_at: -1 }`
- `{ message_id: 1 }` unique if 외부 메시지 id 중복 방지 필요
- `{ expires_at: 1 }` with TTL if 자동 삭제 필요

### 5.2 세션/메시지 분리 이유

메시지를 세션 문서 안의 배열 하나에 계속 쌓는 방식은 초기에 단순하지만, 아래 문제가 생긴다.

- 대화가 길어질수록 한 문서가 과도하게 커진다.
- 일부 메시지 수정/재처리/검색 동기화가 어려워진다.
- 긴 세션 하나 때문에 전체 문서를 매번 다시 써야 한다.

따라서 운영 기준으로는:

- 세션 컬렉션: 목록 조회용
- 메시지 컬렉션: 원문 저장용

구조가 더 안전하다.

### 5.3 보존 정책

MongoDB가 `10GB`이므로, 무제한 보관은 피하는 것이 좋다.

권장안:

- 기본 운영 데이터: `90일` 또는 `180일` 보관
- 장기 보관이 꼭 필요하면 별도 archive 전략 검토
- TTL 삭제 기준은 `expires_at` 필드로 명시 관리

주의:

- MongoDB는 원문 저장소이므로 OpenSearch보다 먼저 삭제되면 안 된다.
- OpenSearch 인덱스 보존 기간은 MongoDB보다 같거나 짧게 맞추는 편이 안전하다.

## 6. OpenSearch 설계

OpenSearch는 "검색을 빠르게 하기 위한 사본"이다.  
MongoDB 전체 필드를 그대로 복제하지 말고, 검색에 필요한 필드만 넣는 것이 좋다.

### 6.1 권장 인덱스

OpenSearch는 역할을 두 갈래로 나누는 것이 좋다.

- 대화 검색 인덱스
- RAG 지식검색 인덱스

둘을 같은 인덱스에 섞지 말고, 목적별로 분리해야 한다.

#### `message_search`

메시지 단위 전문 검색용 인덱스다.

예시 문서:

```json
{
  "message_id": "msg_20260320_000001",
  "conversation_id": "conv_20260320_001",
  "user_id": "u12345",
  "channel": "cube",
  "role": "user",
  "content": "지난달 급여에서 공제된 항목 설명해줘",
  "tags": ["payroll", "hr"],
  "created_at": "2026-03-20T08:10:03Z",
  "expires_at": "2026-09-20T00:00:00Z"
}
```

이 인덱스로 가능한 것:

- 키워드 검색
- 날짜 범위 검색
- 사용자별 검색
- 역할별 검색
- 운영 키워드 집계

#### `conversation_summary`

세션 단위 검색, 목록, 운영 화면 최적화용 인덱스다.

예시 문서:

```json
{
  "conversation_id": "conv_20260320_001",
  "user_id": "u12345",
  "channel": "cube",
  "title": "급여 공제 문의",
  "summary": "사용자가 급여 공제 내역과 계산 기준을 문의함",
  "message_count": 14,
  "tags": ["payroll", "hr"],
  "last_message_at": "2026-03-20T08:12:33Z",
  "status": "open",
  "expires_at": "2026-09-20T00:00:00Z"
}
```

이 인덱스로 가능한 것:

- 세션 목록 검색
- 상담 주제별 필터링
- 운영 대시보드 집계
- 최근 이슈 세션 탐색

### 6.2 OpenSearch에 넣지 말아야 할 것

- 전체 원문 전체를 무조건 중복 저장하는 습관
- 검색에 쓰지 않는 중첩 메타데이터
- 매우 큰 attachment 본문
- 원본 복구에 필요한 유일 데이터

즉 OpenSearch는 검색 용도 필드만 가져가야 한다.

### 6.3 검색 흐름

예시 1: 특정 키워드가 포함된 대화 찾기

1. OpenSearch `message_search`에서 `"공제"` 검색
2. 관련 `conversation_id`, `message_id` 목록 획득
3. 상세 내용은 MongoDB `conversation_messages`에서 조회

예시 2: 특정 사용자의 최근 세션 찾기

1. OpenSearch `conversation_summary`에서 `user_id` + 최근순 정렬
2. 세션 선택
3. MongoDB에서 세션 메시지 원문 조회

즉 OpenSearch는 "찾는 곳", MongoDB는 "읽는 곳"이다.

### 6.4 RAG용 OpenSearch 구조

질문하신 계획처럼 다양한 문서를 RAG로 활용하려면, OpenSearch는 단순 대화 검색을 넘어서 **지식 검색 엔진** 역할을 하게 된다.

이 경우에도 원칙은 같다.

- 원본 문서와 운영 메타데이터의 기준점은 MongoDB 또는 파일 저장소
- OpenSearch는 chunk 단위 검색과 vector 검색 담당

#### 권장 인덱스

##### `knowledge_documents`

문서 단위 메타데이터 인덱스다.

예시 문서:

```json
{
  "document_id": "doc_hr_001",
  "title": "복지포인트 운영 가이드",
  "source_type": "pdf",
  "source_path": "/docs/hr/benefit_guide.pdf",
  "category": "hr",
  "tags": ["복지", "포인트", "인사"],
  "version": 3,
  "updated_at": "2026-03-20T07:00:00Z",
  "is_active": true
}
```

##### `knowledge_chunks`

RAG 검색의 핵심 인덱스다. 문서를 chunk로 쪼개고 임베딩과 함께 저장한다.

예시 문서:

```json
{
  "chunk_id": "doc_hr_001_00012",
  "document_id": "doc_hr_001",
  "title": "복지포인트 운영 가이드",
  "category": "hr",
  "tags": ["복지", "포인트", "인사"],
  "content": "복지포인트는 매년 1월 1일 기준 재직자에게 지급한다.",
  "chunk_order": 12,
  "source_path": "/docs/hr/benefit_guide.pdf",
  "updated_at": "2026-03-20T07:00:00Z",
  "embedding": [0.0123, -0.0311, 0.0045]
}
```

권장 필드:

- `document_id`
- `chunk_id`
- `title`
- `content`
- `category`
- `tags`
- `source_path`
- `updated_at`
- `embedding`

#### 검색 방식

RAG는 보통 아래 방식으로 조회한다.

1. 사용자 질문을 embedding 생성
2. OpenSearch에서 vector similarity 검색
3. 동시에 `content`, `title`, `tags` 기준 keyword/BM25 검색
4. 결과를 hybrid score로 합치거나 rerank
5. 상위 chunk를 LLM context로 주입

즉 OpenSearch는 RAG에서 단순 키워드 검색이 아니라 **hybrid retrieval 계층**이 된다.

### 6.5 대화 검색과 RAG 검색을 분리해야 하는 이유

OpenSearch 하나만 써도 인덱스는 분리하는 것이 좋다.

- `message_search`, `conversation_summary`: 운영 대화 검색
- `knowledge_documents`, `knowledge_chunks`: RAG 검색

분리 이유:

- analyzer와 mapping이 다르다.
- 보존 기간이 다르다.
- access control 기준이 다를 수 있다.
- chunk 검색은 conversation 검색보다 문서 수와 질의 패턴이 다르다.

실무적으로는 "같은 OpenSearch 클러스터, 다른 index namespace" 구조가 가장 관리하기 쉽다.

## 7. Redis 설계

질문하신 방향대로, Redis는 **캐시 용도**, 특히 **향후 CDN과 운영 보조 계층**에 쓰는 것이 맞다.

현재 코드 기준으로도 Redis는 아래 역할에 더 잘 맞는다.

- `api/conversation_service.py`: 최근 대화 캐시
- `api/cdn/cdn_service.py`: CDN 메타데이터 캐시
- `api/cube/queue.py`: 큐
- `api/utils/scheduler/_lock.py`: 분산 락

### 7.1 Redis에 권장하는 역할

#### 최근 대화 캐시

MongoDB에서 읽은 최근 N개 메시지를 Redis에 짧은 TTL로 캐시한다.

예시 키:

- `chat:recent:{conversation_id}`

권장 정책:

- TTL: `10분 ~ 1시간`
- 최근 `20 ~ 50`개 메시지만 저장

이 용도는 LLM 호출 직전 컨텍스트 구성 속도를 높이는 데 효과적이다.

#### CDN 메타데이터 캐시

향후 CDN에 Redis를 쓰려면 다음이 적합하다.

- 파일 메타데이터 캐시
- 변환 이미지 존재 여부 캐시
- pre-signed URL 또는 일회성 토큰 상태 저장
- 다운로드 rate limit
- 업로드/썸네일 작업 상태 저장

즉 Redis는 CDN의 **파일 원본 저장소**가 아니라, **빠른 조회와 상태 관리 계층**으로 써야 한다.

#### RAG 조회 캐시

Redis는 RAG에서도 보조 계층으로 유용하다.

- 자주 들어오는 질문의 retrieval 결과 캐시
- 문서 파이프라인 작업 상태 저장
- embedding 생성 완료 여부 캐시
- ingest dedup 키 저장

다만 Redis에 chunk 본문 전체나 embedding 대량 적재는 권장하지 않는다.

#### 큐 / 락 / 중복방지

현재 구조와 가장 잘 맞는 Redis 사용처다.

- Cube 수신 큐
- 작업 재시도 상태
- 스케줄러 분산 락
- message dedup 키

이 부분은 계속 Redis에 두는 것이 맞다.

### 7.2 Redis에 두지 않는 것이 좋은 것

- 전체 대화 원문 장기 보관
- 모든 세션 전체 이력 상시 적재
- CDN 원본 파일 저장

메모리 `1.5GB`는 큐/락/캐시에는 충분할 수 있지만, 장기 history 저장에는 빠르게 부족해진다.

---

## 8. 데이터 동기화 방식

핵심은 **MongoDB 먼저 저장, OpenSearch 나중 인덱싱**이다.

### 8.1 쓰기 흐름

1. 사용자 메시지 수신
2. MongoDB `conversation_messages`에 메시지 저장
3. MongoDB `conversation_sessions`의 `last_message_at`, `message_count` 갱신
4. Redis 최근 대화 캐시 갱신
5. OpenSearch 인덱싱 작업 enqueue
6. 비동기 worker가 OpenSearch `message_search`, `conversation_summary` 갱신

### 8.2 읽기 흐름

#### 일반 대화 처리

1. Redis 최근 대화 캐시 조회
2. 캐시 miss면 MongoDB 조회
3. 결과를 Redis에 재적재

#### 검색 화면

1. OpenSearch 검색
2. 결과의 `conversation_id`, `message_id` 확보
3. MongoDB에서 상세 원문 조회

### 8.3 장애 허용 기준

- MongoDB 저장 실패: 요청 실패 처리
- Redis 캐시 실패: 로그만 남기고 계속 진행 가능
- OpenSearch 인덱싱 실패: 재시도 큐로 복구

정리하면:

- MongoDB 실패는 치명적
- Redis 실패는 성능 저하
- OpenSearch 실패는 검색 지연

이렇게 계층별 중요도를 분리해야 운영이 단순해진다.

---

## 9. 보유 자원 기준 권장 운영안

현재 보유 자원:

- `Redis`: 1.5GB Memory
- `MongoDB`: 10GB
- `OpenSearch`: 100GB

### 9.1 권장 우선순위

가장 합리적인 배치는 아래와 같다.

#### MongoDB 10GB

- 대화 원문과 세션 메타데이터 저장
- 단, 보존 정책을 반드시 둔다
- 권장 보존 기준: `90일` 시작

10GB는 "무한 보관" 용량은 아니다.  
대화량이 많은 서비스라면 메시지 증가량을 먼저 측정해야 한다.

#### OpenSearch 100GB

- 전문 검색, 운영 분석, RAG 검색에 적극 활용
- MongoDB보다 여유가 크므로 검색 인덱스 보관에 유리
- 단, 인덱스는 원문보다 실제 디스크 사용량이 커질 수 있으므로 필드 최소화가 중요

권장:

- message index와 summary index를 분리
- knowledge document index와 knowledge chunk index를 분리
- analyzer/mapping을 단순하게 유지
- 검색 대상 필드만 넣기

RAG를 같이 운영한다면 OpenSearch 저장 공간은 대화 검색보다 `knowledge_chunks`가 더 많이 차지할 가능성이 높다.  
특히 vector 필드는 chunk 수가 늘수록 비용이 빠르게 증가하므로, chunk 크기와 중복 chunk 수를 초기에 제어해야 한다.

#### Redis 1.5GB

- 최근 대화 캐시
- CDN 메타데이터/토큰 캐시
- 큐/락/dedup

권장:

- 메모리 상한 알림 설정
- TTL 없는 대용량 키 금지
- 큐와 캐시 key prefix 분리

### 9.2 실제 역할 분배 예시

| 영역 | 저장소 | 설명 |
|------|------|------|
| 대화 원문 | MongoDB | 모든 메시지 원본 |
| 세션 목록 | MongoDB | 사용자별 최근 세션 조회 기준 |
| 대화 검색 | OpenSearch | 키워드/조건/기간 검색 |
| RAG 지식검색 | OpenSearch | chunk 검색, hybrid retrieval |
| 운영 집계 | OpenSearch | 키워드 빈도, 상태 집계 |
| 최근 컨텍스트 | Redis | 최근 20~50개 메시지 |
| CDN 메타데이터 캐시 | Redis | 파일 메타, 토큰, 다운로드 상태 |
| RAG 조회 캐시 | Redis | 자주 쓰는 query 결과, ingest 상태 |
| 큐/락 | Redis | 작업 큐, dedup, scheduler lock |

## 10. 현재 코드 기준 마이그레이션 방향

### 10.1 대화 history

현재:

- `api/conversation_service.py`가 Redis 또는 in-memory를 사용
- 키 구조는 `chat:{user_id}`
- 사용자 단위로 메시지 배열 전체를 JSON 저장

권장 변경:

- MongoDB 기반 `conversation_repository` 추가
- Redis는 `recent history cache` 역할만 수행
- 키는 `user_id`보다 `conversation_id` 중심으로 전환

이유:

- 사용자 한 명이 여러 세션을 가질 수 있다.
- 검색/운영/이력 관리 기준은 보통 세션 단위다.

### 10.2 CDN

현재:

- 원본 파일은 파일시스템/PVC 저장
- 메타데이터는 Redis 또는 in-memory

권장:

- 원본 파일: 계속 파일시스템/PVC 또는 향후 object storage
- 영속 메타데이터: MongoDB
- Redis: hot metadata cache, token, variant state

즉 CDN에서도 Redis를 "캐시와 상태 저장"으로만 두는 방향이 맞다.

### 10.3 OpenSearch 연동

권장 방식:

- 앱 요청 경로에서 직접 OpenSearch 동기 호출하지 않기
- MongoDB 저장 후 큐 기반 비동기 인덱싱
- 인덱싱 실패 이벤트는 재시도 가능하도록 message id 기준 멱등 처리

### 10.4 RAG 파이프라인

문서 RAG를 붙일 계획이면 아래 흐름이 가장 무난하다.

1. 원본 문서 수집
2. 문서 메타데이터 저장
3. 텍스트 추출
4. chunk 분할
5. embedding 생성
6. OpenSearch `knowledge_chunks` 인덱싱
7. 질의 시 hybrid retrieval
8. 상위 chunk를 LLM context로 전달

권장 원칙:

- 원본 문서 파일은 파일시스템/PVC 또는 별도 문서 저장소에 둔다.
- 문서 메타데이터와 버전 정보는 MongoDB에 둔다.
- 검색용 chunk와 embedding만 OpenSearch에 둔다.
- Redis는 ingest 상태와 retrieval cache만 담당한다.

## 11. 권장 구현 순서

### 1단계

- MongoDB에 대화 세션/메시지 저장소 도입
- Redis history 저장 제거
- Redis recent cache 추가

### 2단계

- OpenSearch `message_search` 인덱스 추가
- 검색 API 또는 운영 조회 기능 추가

### 3단계

- `conversation_summary` 인덱스 추가
- 운영 대시보드/세션 검색 최적화

### 4단계

- CDN 메타데이터의 MongoDB 영속화 검토
- Redis는 CDN 캐시와 작업 상태 전용으로 정리

## 12. 최종 권장안

질문의 요지를 기준으로 정리하면 다음이 가장 적합하다.

- **MongoDB를 메인 DB로 사용**
- **OpenSearch는 검색 엔진이자 RAG retrieval 계층으로 사용**
- **Redis는 cache + queue + lock + CDN/RAG 보조 상태 저장으로 사용**

특히 Redis는 질문하신 대로 **나중에 CDN을 위한 cache 용도**로 두는 방향이 맞다.  
다만 "CDN 파일 자체"를 Redis에 두는 것이 아니라, 아래에 쓰는 것이 맞다.

- 파일 메타데이터 캐시
- 이미지 variant 존재 여부 캐시
- 다운로드 토큰 / 임시 접근 제어
- 업로드/썸네일 비동기 작업 상태
- rate limit / dedup

OpenSearch는 앞으로 두 역할을 맡게 된다.

- 대화 history 검색
- 문서 RAG용 hybrid retrieval

대화 저장 관점에서는 다음 규칙만 지키면 된다.

- 원본은 MongoDB
- 검색과 RAG retrieval은 OpenSearch
- 속도 최적화는 Redis

이 구조가 현재 자원 구성과 가장 잘 맞는다.
