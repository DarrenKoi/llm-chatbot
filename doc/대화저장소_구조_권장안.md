# 대화 저장소 구조 권장안

## 1. 목적

현재 프로젝트는 `api/conversation_service.py`에서 Redis를 이용해 사용자별 대화 history를 저장하고 있다.
앞으로 사용할 수 있는 저장소가 `Redis(1.5GB Memory)`, `MongoDB(10GB)`, `OpenSearch(100GB)`라면, 현재 요구사항 기준으로는 아래 역할 분리가 가장 단순하고 안정적이다.

- `MongoDB`: 대화 원문 저장소, 정합성 기준, 요약 저장소
- `Redis`: CDN 메타데이터 캐시 및 기타 단기 상태 저장
- `OpenSearch`: 향후 검색/RAG가 필요해질 때 붙이는 선택 계층

이 문서는 현재 Cube 식별자 구조와 운영 방식에 맞춰, 대화 이력을 어떻게 저장하고 어떻게 불러오며 어떻게 오래된 이력을 압축할지 설명한다.

---

## 2. 결론

### 권장 역할 분리

| 저장소 | 권장 역할 | 현재 기준 비권장 역할 |
|------|------|------|
| MongoDB | 대화 원문 저장, 최근 이력 조회, 요약 저장, 보존 정책 관리 | 없음 |
| Redis | CDN 메타데이터 캐시, 토큰/상태성 데이터 | 대화 history 캐시, 장기 대화 보관 |
| OpenSearch | 향후 대화 검색, RAG retrieval | 현재 시점의 필수 저장소 역할 |

### 한 줄 요약

- **대화 원문은 MongoDB에 전부 저장**
- **LLM 호출 시에는 최근 N개만 MongoDB에서 직접 조회**
- **오래된 이력은 스케줄러가 LLM으로 요약 압축**
- **Redis는 대화 경로에서 빼고 CDN 용도로만 사용**

---

## 3. 왜 이 구조가 맞는가

### 3.1 Cube 식별자 기준으로 보면 MongoDB 단독 구조가 자연스럽다

현재 Cube에서 전달받는 주요 식별자는 다음과 같다.

- `username`: 중복 가능
- `uniquename`: 사용자 고유값, `str(int)`
- `channelid`: 사용자가 방을 떠나기 전까지 유지되는 장수 room id
- `messageid`: 증가하는 `str(int)`
- `edms_id`: UUID 형태로 보이는 delivery id

이 조합이면 대화 원문 저장에는 별도 캐시 계층 없이도 충분하다.

- 사용자 식별 키는 `uniquename`을 사용하면 된다.
- 대화방 식별 키는 `channelid`를 사용하면 된다.
- 메시지 순서와 dedup 판단에는 `messageid`와 `edms_id`를 활용하면 된다.

즉, 현재 단계에서는 `uniquename + channelid` 기준으로 원문을 저장하고, 조회 시 최근 N개만 가져오는 단순 구조가 가장 현실적이다.

### 3.2 왜 Redis 대화 캐시가 지금은 필요하지 않은가

현재 서비스는 사용자가 타이핑을 하고, 서버가 답을 생성하는 일반적인 챗 인터랙션이다.
이 경우 요청 빈도는 초당 다량 호출되는 시스템이 아니라 사용자 입력 속도에 의해 제한된다.

따라서 현재 기준으로는 아래 쪽이 더 중요하다.

- 구조 단순성
- 운영 복잡도 감소
- 원본 저장소 일원화
- 캐시 정합성 문제 제거

MongoDB에서 최근 20~50개 메시지를 읽는 비용은 보통 충분히 감당 가능하다.
반면 Redis 캐시를 끼우면 다음 문제가 추가된다.

- cache miss / refill 로직
- stale cache 관리
- user turn 저장 후 assistant turn 실패 시 정합성 처리
- MongoDB와 Redis 이중 운영

따라서 성능 병목이 실제로 관찰되기 전까지는, 대화 history용 Redis는 넣지 않는 편이 좋다.

### 3.3 왜 `channelid`만으로 전체 history를 그대로 쓰면 안 되는가

`channelid`는 장수 room id이므로, 저장 키로 쓰는 것은 괜찮다.
다만 LLM에 넣는 runtime history를 그 room의 전체 누적 이력으로 잡으면 안 된다.

문제는 저장이 아니라 조회 범위다.

- 저장은 전체 원문을 모두 해도 된다.
- 조회는 최근 N개만 가져와야 한다.

즉 아래 방식은 적절하다.

- MongoDB에는 `uniquename + channelid` 기준으로 모든 메시지를 저장
- LLM 호출 직전에는 최근 20~50개만 조회

걱정해야 할 것은 "무제한 저장"이 아니라 "무제한 조회"다.

## 4. 목표 아키텍처

```text
Cube / API Client
    |
    v
Flask App
    |
    +-- 1) MongoDB 원문 저장
    |       - conversation_messages
    |       - conversation_summaries
    |
    +-- 2) MongoDB 최근 이력 조회
    |       - uniquename + channelid 기준 최근 N개
    |
    +-- 3) Scheduler 기반 요약 압축
    |       - 오래된 메시지 구간 summary 생성
    |
    +-- 4) Redis
            - CDN metadata cache
            - 기타 단기 상태성 데이터
```

### 처리 원칙

1. 사용자 메시지는 먼저 MongoDB에 저장한다.
2. LLM 호출 시에는 MongoDB에서 최근 N개만 직접 조회한다.
3. 오래된 메시지는 스케줄러가 비동기로 요약한다.
4. 원문은 그대로 유지하고, summary는 파생 데이터로만 다룬다.
5. Redis는 대화 경로에서 제거하고 CDN 경로에만 사용한다.

## 5. MongoDB 설계

현재 요구사항 기준으로는 세션 컬렉션까지 먼저 만들기보다, `메시지 컬렉션 + 요약 컬렉션`으로 시작하는 편이 더 단순하다.

### 5.1 권장 컬렉션

#### `conversation_messages`

실제 메시지 원문을 한 건씩 저장한다.

예시 필드:

```json
{
  "_id": "mongo-object-id",
  "uniquename": "1234567",
  "username": "홍길동",
  "channelid": "room-001",
  "messageid": "8923441",
  "edms_id": "c523c7043-b123-4532-9gds1-ewtkjw3",
  "role": "user",
  "content": "지난달 급여에서 공제된 항목 설명해줘",
  "metadata": {
    "source": "cube"
  },
  "created_at": "2026-03-23T08:10:03Z",
  "expires_at": "2026-09-23T00:00:00Z"
}
```

권장 인덱스:

- `{ uniquename: 1, channelid: 1, created_at: -1 }`
- `{ uniquename: 1, channelid: 1, messageid: 1 }`
- `{ edms_id: 1 }` unique if 실제로 delivery 기준 유일성이 보장된다면 사용
- `{ expires_at: 1 }` with TTL if 자동 삭제 필요

권장 원칙:

- 사용자 식별 기준은 `username`이 아니라 `uniquename`
- 조회 기준은 `uniquename + channelid`
- dedup 기준은 `messageid` 또는 `edms_id`

#### `conversation_summaries`

오래된 메시지 구간을 LLM으로 요약한 결과를 저장한다.

예시 필드:

```json
{
  "_id": "summary_20260323_001",
  "uniquename": "1234567",
  "channelid": "room-001",
  "start_messageid": "8923000",
  "end_messageid": "8923099",
  "message_count": 100,
  "from_at": "2026-01-01T09:00:00Z",
  "to_at": "2026-01-15T18:00:00Z",
  "summary_text": "사용자는 급여 공제, 복지포인트, 휴가 정책을 순차적으로 문의했다.",
  "model": "gpt-5.4",
  "summary_version": 1,
  "created_at": "2026-03-23T09:00:00Z"
}
```

권장 인덱스:

- `{ uniquename: 1, channelid: 1, end_messageid: -1 }`
- `{ uniquename: 1, channelid: 1, from_at: 1 }`
- `{ uniquename: 1, channelid: 1, start_messageid: 1, end_messageid: 1 }`

### 5.2 왜 메시지 컬렉션 하나로 먼저 시작하는가

현재 서비스는 아래 요구가 먼저 중요하다.

- 원문 저장
- 최근 이력 조회
- dedup 가능성 확보
- 요약 압축 연결

이 단계에서는 별도 `conversation_id`나 `conversation_sessions`를 먼저 만들지 않아도 된다.
장수 room 구조에서 실제 runtime 조회 기준은 `uniquename + channelid`이고, LLM에는 최근 N개만 넣으면 되기 때문이다.

향후 아래 요구가 강해지면 세션 컬렉션을 추가하면 된다.

- 상담 이력 목록 화면
- 대화 단위 운영 검색
- 세션 상태 관리
- 세션 제목/태그/상태 저장

### 5.3 보존 정책

MongoDB가 `10GB`이므로, 무제한 보관 여부는 운영 기준에 맞춰 정해야 한다.

권장안:

- 원문은 기본 `90일` 또는 `180일` 보관
- 더 길게 보관해야 하면 archive 전략 별도 검토
- 삭제 기준은 `expires_at` 필드로 명시 관리

중요한 원칙:

- summary가 생겨도 raw message를 즉시 지우지 않는다.
- summary는 원문의 대체물이 아니라 파생 데이터다.

## 6. 읽기 전략

### 6.1 LLM 호출용 최근 이력 조회

실시간 답변 생성 시에는 아래처럼 최근 이력만 가져온다.

예시 조회 조건:

```text
find({ uniquename, channelid })
sort(created_at desc)
limit(50)
```

이후 애플리케이션에서 시간순으로 다시 정렬해 LLM에 전달한다.

권장 범위:

- 최근 `20 ~ 50`개 메시지
- 필요 시 role 기준 필터 포함

핵심은 전체 room history를 조회하지 않는 것이다.

### 6.2 오래된 이력 활용 방식

오래된 이력은 raw message 전체를 다시 LLM에 넣지 않고, 저장된 summary를 활용한다.

즉 runtime context는 다음 형태가 된다.

- 오래된 구간 summary 1개 이상
- 최신 raw message 20~50개

이 구조면 맥락을 유지하면서도 context 크기를 통제할 수 있다.

## 7. 요약 압축 전략

### 7.1 요약은 Scheduler가 수행한다

summary compaction은 request path가 아니라 scheduler가 수행하는 background 작업으로 두는 것이 맞다.

이유:

- 사용자 응답 지연과 분리 가능
- 긴 history 처리 비용을 별도 제어 가능
- 재시도와 운영 관리가 쉬움

### 7.2 어떤 메시지를 요약할 것인가

달력 기준만으로 자르기보다, "최신 active tail은 raw로 남기고 그 이전 구간을 요약"하는 방식이 좋다.

권장안:

- 최신 50개 메시지는 raw로 유지
- 그보다 오래된 메시지는 100개 단위 같은 bounded chunk로 요약
- 또는 30일 이상 지난 구간 중 최신 tail 밖에 있는 것만 요약

핵심은 매번 방 전체를 재요약하지 않는 것이다.

### 7.3 요약 원칙

1. 증분 방식으로 요약한다.
2. 이미 요약된 구간은 다시 처리하지 않는다.
3. 아직 활발히 이어지는 최신 메시지 구간은 요약하지 않는다.
4. summary는 재생성 가능해야 한다.
5. raw message는 source of truth로 유지한다.

### 7.4 요약 스케줄러 흐름

```text
Scheduler
   |
   +-- 요약 대상 room-user 범위 탐색
   +-- 아직 요약되지 않은 오래된 메시지 구간 선택
   +-- LLM으로 summary 생성
   +-- conversation_summaries 저장
   +-- 필요 시 summary coverage 마킹
```

### 7.5 요약 시 필요한 메타데이터

요약 문서에는 최소한 아래 값이 있어야 한다.

- `uniquename`
- `channelid`
- `start_messageid`
- `end_messageid`
- `message_count`
- `from_at`
- `to_at`
- `summary_text`
- `model`
- `summary_version`

이 값이 있어야 중복 요약 방지와 coverage 추적이 가능하다.

## 8. Redis 설계

현재 요구사항 기준으로 Redis는 대화 history 경로에서 제거하는 것이 맞다.

### 8.1 Redis에 권장하는 역할

- CDN 메타데이터 캐시
- 다운로드/업로드 상태 저장
- 일회성 토큰, rate limit
- 기타 짧은 TTL의 상태성 데이터

### 8.2 Redis에 넣지 않는 것이 좋은 것

- 대화 원문 장기 보관
- 최근 대화 history 캐시
- summary 저장

즉 현재 기준 Redis는 CDN 보조 계층으로 한정하는 편이 가장 단순하다.

---

## 9. OpenSearch 설계

OpenSearch는 지금 당장 붙일 필요는 없다.
향후 아래 요구가 생기면 도입한다.

- 운영자용 대화 검색
- 키워드/조건 검색
- 대화 집계
- 문서 기반 RAG retrieval

### 9.1 도입 시 원칙

- MongoDB가 원본 저장소
- OpenSearch는 검색용 파생 인덱스
- 검색 실패가 원문 저장 실패로 이어지지 않게 설계

### 9.2 향후 권장 인덱스

- `message_search`: 메시지 단위 검색
- `knowledge_documents`: 문서 메타데이터
- `knowledge_chunks`: RAG chunk 검색

현재 단계에서는 문서에만 방향성으로 남기고, 구현 우선순위에서는 뒤로 미루는 것이 맞다.

---

## 10. 최종 권장안

현재 프로젝트와 Cube 식별자 구조를 기준으로 한 최종 권장안은 아래와 같다.

1. 대화 원문은 MongoDB `conversation_messages`에 모두 저장한다.
2. 저장 키는 `uniquename + channelid + messageid` 계열을 사용한다.
3. dedup 판단은 `messageid` 또는 `edms_id`를 사용한다.
4. 실시간 LLM 호출 시에는 최근 20~50개만 MongoDB에서 직접 조회한다.
5. 오래된 메시지는 scheduler가 LLM으로 요약해 `conversation_summaries`에 저장한다.
6. runtime context는 `older summaries + latest raw messages` 조합으로 만든다.
7. Redis는 대화 경로에서 제거하고 CDN 용도로만 사용한다.
8. OpenSearch는 실제 검색/RAG 요구가 생길 때 추가한다.

### 한 줄 정리

**지금은 MongoDB 단독 대화 저장 구조가 맞고, 오래된 이력 압축은 scheduler 기반 summary compaction으로 푸는 것이 가장 실용적이다.**
