# 아키텍처

## 현재 구현 범위

이 저장소는 사내 메신저 Cube와 연동되는 Flask 서버로, 현재 아래 기능이 실제 구현되어 있다.

- Cube 웹훅 수신 및 페이로드 파싱
- 사용자별 대화 이력 저장
- OpenAI 호환 LLM API 호출
- Cube `multiMessage` API로 응답 전송
- CDN 파일 업로드, 원본 다운로드, 이미지 리사이즈/썸네일 제공
- 구조화 activity 로그 기록
- APScheduler 기반 백그라운드 작업 등록
- Redis 분산 락 기반 스케줄러 중복 실행 방지

Tool calling이나 별도 업무 도구 오케스트레이션은 아직 이 저장소에 구현되어 있지 않다.

## 요청 처리 흐름

현재 Cube 메시지 처리는 동기식으로 수행된다. 즉, 라우터가 `202 Accepted`를 반환하기 전에 LLM 응답 생성과 Cube 전송까지 끝난다.

```text
1. Cube → POST /api/v1/cube/receiver
2. `api.cube.router.receive_cube()`가 JSON payload를 수신
3. `api.cube.service.handle_cube_message()`가 입력 검증
4. wake-up 메시지(`message_id="-1"` and `!@#...`)면 즉시 스킵
5. `conversation_service.get_history()`로 기존 대화 조회
6. 사용자 메시지를 대화 이력에 append
7. `llm.service.generate_reply()`로 OpenAI 호환 `/chat/completions` 호출
8. LLM 응답을 대화 이력에 append
9. `cube.client.send_multimessage()`로 Cube에 답변 전송
10. activity log / request log 기록
11. `{"status":"accepted","message_id":"..."}` + HTTP 202 반환
```

오류 흐름은 다음과 같이 정리된다.

- 잘못된 JSON 또는 Cube 필드 누락: `400`
- LLM 호출 실패: `502`
- Cube 응답 전송 실패: `502`

## 애플리케이션 계층

### 1. 앱 초기화 계층

`api/__init__.py`

- `setup_logging()`으로 로그 핸들러 초기화
- `start_scheduler()`로 APScheduler 시작
- `/` 샘플 페이지 라우트 제공
- `discover_blueprints()`로 `router*.py` 자동 등록
- 모든 요청에 대해 `request_id`, 처리 시간, 사용자 ID를 activity log에 남김

### 2. Cube 도메인 계층

`api/cube/`

- `router.py`: Cube 엔드포인트 공개
- `service.py`: Cube 메시지 처리 오케스트레이션
- `payload.py`: richnotificationmessage / 평탄 payload 파싱, Cube 전송 payload 생성
- `client.py`: Cube `multiMessage`, `richnotification` HTTP 호출
- `models.py`: `CubeIncomingMessage`, `CubeHandledMessage` dataclass 정의

현재 실제 응답 전송 경로는 `send_multimessage()`다. `send_richnotification()` 빌더와 클라이언트는 준비되어 있지만 기본 처리 경로에는 연결되어 있지 않다.

### 3. LLM 계층

`api/llm/service.py`

- OpenAI 호환 API 형식 사용
- `LLM_SYSTEM_PROMPT`를 system message로 주입
- 저장된 대화 이력 중 `user`, `assistant`, `system` role만 전달
- 응답 `choices[0].message.content`가 문자열 또는 list인 경우 모두 정규화

### 4. 대화 이력 계층

`api/conversation_service.py`

- Redis 우선, 실패 시 In-Memory 폴백
- 사용자별 키 `chat:{user_id}` 사용
- `CONVERSATION_MAX_MESSAGES`만 유지
- Redis 사용 시 `CONVERSATION_TTL_SECONDS` TTL 적용
- In-Memory 사용 시 `OrderedDict` 기반 LRU 방식으로 최대 1,000명 사용자 유지

### 5. CDN 계층

`api/cdn/`

- `POST /api/v1/cdn/upload`
- `GET /cdn/files/<file_id>`
- 지원 확장자: 이미지(`png`, `jpg`, `jpeg`, `gif`, `webp`)와 문서(`xlsx`, `pptx`, `docx`)
- 이미지에 대해 `w`, `h`, `thumbnail` 쿼리로 variant 생성
- 메타데이터는 Redis 또는 In-Memory에 저장
- 파일 원본/변형은 `CDN_STORAGE_DIR` 하위 파일시스템에 저장

### 6. 로그 계층

`api/utils/logger/`

- stdout 운영 로그 + JSONL activity 로그를 함께 구성
- activity 로그 기본 위치: `logs/activity/activity.jsonl`
- `request_id`, HTTP 메서드, 경로, 사용자 ID 등 요청 컨텍스트 자동 포함
- 주제별/테마별 logger를 별도 파일로 분리 가능
- `TimedRotatingFileHandler` 기반 일 단위 로그 로테이션

### 7. 스케줄러 계층

`api/utils/scheduler/`

- `BackgroundScheduler`를 앱 시작 시 1회만 기동
- `tasks/` 하위 모듈 자동 탐색
- `@scheduled_job` 데코레이터 또는 legacy `register()` 방식 지원
- Redis 기반 분산 락으로 멀티 uWSGI worker 환경의 중복 실행 방지
- 현재 등록된 운영 작업은 `cleanup_uwsgi_logs_job()` 1개이며, 매일 01:00에 7일 초과 uWSGI 로그 파일을 정리한다

## 저장소 및 외부 의존성

### Redis 사용 방식

| 영역 | 설정 | 동작 |
|------|------|------|
| 대화 이력 | `REDIS_URL`, `REDIS_FALLBACK_URL` | primary → fallback 순서로 연결 시도, 실패 시 In-Memory |
| CDN 메타데이터 | `CDN_REDIS_URL` | 실패 시 In-Memory |
| 스케줄러 락 | `SCHEDULER_REDIS_URL` | Redis 미가용 시 잡 실행 자체를 스킵 |

### 파일시스템 사용 방식

| 영역 | 기본 경로 | 설명 |
|------|-----------|------|
| 로그 | `./logs` | activity 로그, uWSGI 로그 저장 |
| PVC 루트 | `/project/workSpace/pvc/download` | Linux 배포 기본값 |
| CDN 파일 | `<PVC_ROOT>/cdn/files` | 원본 및 variant 저장 |

## 배포/런타임 구조

`wsgi.ini` 기준 런타임은 다음과 같다.

- uWSGI `workers = 3`
- `threads = 2`
- `lazy-apps = true`
- `thunder-lock = true`
- `touch-reload = /project/workSpace/restart.txt`
- uWSGI 데몬 로그: `./logs/uwsgi-YYYY-MM-DD.log`

멀티 워커 환경에서도 스케줄러 잡이 중복 실행되지 않도록 Redis 락을 반드시 함께 사용한다.

## 현재 상태 요약

현재 아키텍처는 "Cube 수신 → 대화 이력 반영 → LLM 호출 → Cube 재전송"의 최소 동작 루프와, 이를 운영하기 위한 CDN/로그/스케줄러 기반 기능까지 포함한 상태다. 다만 Tool calling, 업무 시스템 연동, 비동기 작업 큐, CDN 만료 파일 자동 삭제 같은 확장 기능은 아직 후속 구현 영역으로 남아 있다.
