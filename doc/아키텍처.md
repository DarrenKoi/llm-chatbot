## 요청 처리 흐름

사용자가 Cube에서 메시지를 보내면 다음과 같은 흐름으로 처리됩니다.

```
1. Cube 웹훅 → POST /api/v1/cube/receiver
2. 사용자 메시지 파싱 (`user`, `message_id`, `channel`, `message`)
3. Redis에서 대화 기록 조회
4. 즉시 202 Accepted 응답 (비동기 처리)
5. 백그라운드 스레드에서 LLM 호출
6. (필요 시) Tool Use 루프 실행
7. 최종 응답을 Cube 리치 알림으로 전송
8. MongoDB에 요청 로그 기록
```

### 비동기 처리 구조

Cube 메신저의 웹훅 특성상, 사용자 요청을 즉시 수락(`202 Accepted`)한 뒤 `ThreadPoolExecutor`로 백그라운드 처리합니다. LLM 응답이 준비되면 Cube API를 통해 결과를 push 방식으로 전달합니다.

```python
# routes.py — 비동기 처리 핵심 로직
executor = ThreadPoolExecutor(max_workers=config.MAX_WORKERS)

@bp.route("/api/v1/cube/receiver", methods=["POST"])
def receive_cube():
    # ... 메시지 파싱 ...
    executor.submit(_process_llm_request, user_id, channel_id, message_id, history, user_msg)
    return jsonify({"status": "accepted"}), 202
```

## 핵심 모듈 구조

```
llm_chatbot/
├── index.py                    # 앱 진입점
├── wsgi.ini                    # uWSGI 설정
├── api/
│   ├── __init__.py             # Flask 앱 팩토리 (create_application)
│   ├── blueprint_loader.py     # router 자동 탐색/등록
│   ├── config.py               # 환경 변수 기반 설정
│   ├── conversation_service.py # 대화 기록 관리 (Redis/In-Memory)
│   ├── cdn/
│   │   ├── __init__.py         # CDN 서비스 export
│   │   ├── cdn_service.py      # 파일 저장/조회 로직
│   │   └── router.py           # CDN 라우트
│   ├── cube/
│   │   └── __init__.py         # Cube 도메인 패키지
│   ├── llm/
│   │   ├── __init__.py         # LLM 서비스 export
│   │   ├── llm_service.py      # LLM 호출 + Tool Use 루프
│   │   └── tools/
│   │       ├── __init__.py     # Tool 정의 및 실행 디스패처
│   │       ├── query_data.py   # 데이터 조회 Tool
│   │       └── create_chart.py # 차트 생성 Tool
│   └── utils/
│       ├── logger/             # 구조화 로그 유틸리티
│       └── scheduler/          # 스케줄러 및 작업 등록
└── tests/
    └── ...                     # pytest 기반 테스트
```

### 모듈별 역할

| 모듈 | 역할 |
|------|------|
| `cdn/router.py` | CDN 업로드/다운로드 엔드포인트 정의 및 Blueprint export |
| `llm/llm_service.py` | LLM API 호출 및 Tool Use 루프 관리 (최대 5회 반복) |
| `conversation_service.py` | 사용자별 대화 기록 저장·조회 (Redis 또는 In-Memory 자동 전환) |
| `blueprint_loader.py` | `api` 하위의 `router*.py`를 자동 탐색해 Blueprint 등록 |
| `llm/tools/__init__.py` | Tool 스키마 정의 및 실행 디스패처 (이름 → 함수 매핑) |

## Tool Use 아키텍처

LLM이 사용자 질문에 답하기 위해 외부 도구가 필요하다고 판단하면, 자율적으로 Tool을 호출합니다.

```
사용자: "지난달 제품별 매출 차트 그려줘"

[LLM 1차 호출]
  → LLM 판단: 데이터가 필요함 → query_data Tool 호출

[Tool 실행: query_data]
  → 결과: {Product A: 1200, Product B: 850, Product C: 2100}

[LLM 2차 호출]
  → LLM 판단: 차트를 그릴 수 있음 → create_chart Tool 호출

[Tool 실행: create_chart]
  → 결과: chart_image_url

[LLM 3차 호출]
  → LLM 판단: 모든 정보 확보 → 최종 텍스트 응답 생성

[Cube로 전송]
  → 텍스트 + 차트 이미지 리치 알림
```

### 현재 제공 Tool

| Tool | 기능 | 파라미터 |
|------|------|---------|
| `query_data` | 데이터베이스/Elasticsearch 조회 | `query` (자연어), `source` (database/elasticsearch) |
| `create_chart` | 차트 이미지 생성 (bar/line/pie) | `chart_type`, `title`, `data` (labels + values) |

### 확장 방법

새로운 Tool을 추가하려면:

1. `api/llm/tools/` 에 `execute(**kwargs) -> str` 함수를 가진 모듈 생성
2. `api/llm/tools/__init__.py`의 `TOOL_DEFINITIONS`에 스키마 추가
3. `TOOL_EXECUTORS` 딕셔너리에 실행 함수 등록

별도의 라우트 변경이나 LLM 프롬프트 수정 없이 Tool만 추가하면 LLM이 자동으로 인식하고 사용합니다.

## 대화 기록 관리

사용자별 대화 맥락을 유지하여 멀티턴 대화를 지원합니다.

| 설정 | 기본값 | 설명 |
|------|--------|------|
| `CONVERSATION_MAX_MESSAGES` | 20 | 사용자당 최대 보관 메시지 수 |
| `CONVERSATION_TTL_SECONDS` | 3600 | 대화 기록 만료 시간 (1시간) |

### 백엔드 자동 전환

| 환경 | 백엔드 | 조건 |
|------|--------|------|
| 운영 | Redis | `REDIS_URL` 환경 변수가 설정된 경우 |
| 개발 | In-Memory (OrderedDict) | `REDIS_URL`이 비어있는 경우 자동 폴백 |

In-Memory 백엔드는 LRU 방식으로 최대 1,000명의 사용자 기록을 유지합니다.

## 로깅 및 모니터링

모든 요청은 MongoDB `request_logs` 컬렉션에 기록됩니다.

```json
{
  "user_id": "사용자 ID",
  "channel_id": "채널 ID",
  "message_id": "Cube 메시지 ID",
  "user_message": "사용자 메시지",
  "reply_text": "LLM 응답",
  "model": "kimi-k2.5",
  "status": "success | error",
  "error": null,
  "total_duration_ms": 2340,
  "llm_calls": [
    {"duration_ms": 1200, "prompt_tokens": 512, "completion_tokens": 128, "tool_calls": ["query_data"]}
  ],
  "tool_executions": [
    {"name": "query_data", "args": {...}, "duration_ms": 45, "success": true}
  ],
  "created_at": "2026-03-02T09:00:00Z"
}
```

LLM 호출 횟수, 토큰 사용량, Tool 실행 시간, 성공/실패 여부까지 상세하게 추적할 수 있습니다.

## 설정 관리

모든 설정은 환경 변수로 관리되며, 로컬 개발 시 `.env` 파일로 오버라이드합니다.

| 환경 변수 | 용도 | 기본값 |
|-----------|------|--------|
| `LLM_BASE_URL` | LLM 서버 주소 | `http://localhost:8000/v1` |
| `LLM_MODEL` | 사용 모델명 | `kimi-k2.5` |
| `LLM_SYSTEM_PROMPT` | 시스템 프롬프트 | `You are a helpful assistant.` |
| `REDIS_URL` | Redis 접속 URL | (빈 값 → In-Memory) |
| `MONGO_URL` | MongoDB 접속 URL | `mongodb://localhost:27017` |
| `CUBE_API_URL` | Cube API 주소 | (빈 값) |
| `CUBE_API_TOKEN` | Cube API 인증 토큰 | (빈 값) |
| `MAX_WORKERS` | 백그라운드 스레드 수 | `4` |
