# 아키텍처

## 전체 시스템 구성

```
  사용자
    │
    ▼
┌─────────────┐    HTTP POST     ┌───────────────────────────────┐
│  Cube 메신저  │ ──────────────▶ │  LLM Chatbot 서버              │
│             │                  │  (Flask + uWSGI)              │
└─────────────┘                  │                               │
                                 │  cube/router.py              │
                                 │     └─ conversation_service.py │
                                 │                               │
                                 │  cdn/router.py               │
                                 │  utils/logger/               │
                                 └──────────────┬───────────────┘
                                                │
                                         ┌──────▼──────┐
                                         │   Redis      │
                                         │  대화 기록    │
                                         └─────────────┘
```

Tool calling 및 실제 응답 생성은 이 저장소가 아닌 별도 저장소에서 담당합니다.

---

## 요청 처리 흐름

사용자가 Cube에서 메시지를 보내면 다음과 같은 흐름으로 처리됩니다.

```
1. Cube 웹훅 → POST /api/v1/cube/receiver
2. `richnotificationmessage.header.from` / `process.processdata` 에서 사용자·채널·메시지 정보 파싱
3. 사용자 메시지를 대화 이력 저장소(Redis 또는 In-Memory)에 기록
4. 구조화 로그에 수신 내역 기록
5. 즉시 202 Accepted 응답 반환
```

## 핵심 모듈 구조

```
llm_chatbot/
├── index.py                    # 앱 진입점
├── wsgi.ini                    # uWSGI 설정
├── api/
│   ├── __init__.py             # Flask 앱 팩토리 (create_application)
│   ├── blueprint_loader.py     # router 자동 탐색/등록
│   ├── config.py               # 환경 변수 기반 설정
│   ├── conversation_service.py # 대화 기록 관리 (Redis/In-Memory)
│   ├── cdn/
│   │   ├── __init__.py         # CDN 서비스 export
│   │   ├── cdn_service.py      # 파일 저장/조회 로직
│   │   └── router.py           # CDN 라우트
│   ├── cube/
│   │   ├── __init__.py         # Cube 도메인 패키지
│   │   └── router.py           # Cube 웹훅 라우트
│   └── utils/
│       ├── logger/             # 구조화 로그 유틸리티
│       └── scheduler/          # 스케줄러 및 작업 등록
└── tests/
    └── ...                     # pytest 기반 테스트
```

### 모듈별 역할

| 모듈 | 역할 |
|------|------|
| `cdn/router.py` | CDN 업로드/다운로드 엔드포인트 정의 및 Blueprint export |
| `cube/router.py` | Cube 웹훅 수신, 요청 필드 파싱, 사용자 메시지 저장 |
| `conversation_service.py` | 사용자별 대화 기록 저장·조회 (Redis 또는 In-Memory 자동 전환) |
| `blueprint_loader.py` | `api` 하위의 `router*.py`를 자동 탐색해 Blueprint 등록 |
| `utils/logger/` | 요청/이벤트 로그 기록 및 파일 로테이션 관리 |

## 대화 기록 관리

사용자별 대화 맥락을 유지하여 멀티턴 대화를 지원합니다.

| 설정 | 기본값 | 설명 |
|------|--------|------|
| `CONVERSATION_MAX_MESSAGES` | 20 | 사용자당 최대 보관 메시지 수 |
| `CONVERSATION_TTL_SECONDS` | 3600 | 대화 기록 만료 시간 (1시간) |

### 백엔드 자동 전환

| 환경 | 백엔드 | 조건 |
|------|--------|------|
| 운영 | Redis | `REDIS_URL` 환경 변수가 설정된 경우 |
| 개발 | In-Memory (OrderedDict) | `REDIS_URL`이 비어있는 경우 자동 폴백 |

In-Memory 백엔드는 LRU 방식으로 최대 1,000명의 사용자 기록을 유지합니다.

## 로깅 및 모니터링

Cube 수신 요청은 `log_request()`와 activity logger를 통해 구조화 로그로 남깁니다.

```json
{
  "user_id": "사용자 ID",
  "user_name": "사용자 이름",
  "channel_id": "채널 ID",
  "message_id": "Cube 메시지 ID",
  "user_message": "사용자 메시지",
  "status": "accepted",
  "processor": "external"
}
```

응답 생성은 별도 저장소에서 수행되므로, 이 저장소에서는 수신 및 저장 상태를 중심으로 추적합니다.

## 설정 관리

모든 설정은 환경 변수로 관리되며, 로컬 개발 시 `.env` 파일로 오버라이드합니다.

| 환경 변수 | 용도 | 기본값 |
|-----------|------|--------|
| `REDIS_URL` | Redis 접속 URL | (빈 값 → In-Memory) |
| `REDIS_FALLBACK_URL` | 보조 Redis 접속 URL | (빈 값) |
| `CONVERSATION_MAX_MESSAGES` | 사용자당 최대 보관 메시지 수 | `20` |
| `CONVERSATION_TTL_SECONDS` | 대화 기록 TTL | `3600` |
| `CDN_BASE_URL` | CDN 다운로드 기준 URL | `http://localhost:5000/cdn/files` |
| `LOG_DIR` | 로그 저장 루트 경로 | `<repo>/logs` |
