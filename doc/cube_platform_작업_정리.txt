Cube 플랫폼 작업 정리

이 문서는 현재 저장소 기준으로 Cube 플랫폼 연동에서 우리가 구현한 내용만 간단히 정리한 메모다.
상세 설계와 변경 이력은 아래 문서와 코드 파일을 함께 보면 된다.

1. 전체 요약

- 이 프로젝트는 Cube 메신저에서 들어온 메시지를 받아 Flask 서버에서 수신하고, Redis 큐를 거쳐 worker가 실제 응답을 생성한 뒤 Cube로 다시 전달하는 구조로 정리되어 있다.
- 초기 목표는 "Cube 안에서 바로 사내 LLM 챗봇을 쓸 수 있게 만들되, 웹훅 ACK 지연과 중복 처리 문제를 줄이고, 운영 상태를 확인할 수 있게 하는 것"이었다.
- 현재 Cube 계층은 thin transport 역할에 가깝고, 실제 답변 생성은 workflow 계층이 담당한다.

2. Cube 수신 경로에서 한 일

- Cube webhook 엔드포인트를 `/api/v1/cube/receiver`로 고정하고, 수신 진입점은 `api/cube/router.py`에서 관리한다.
- Cube payload 형식과 일반 평탄 payload 형식을 모두 읽을 수 있게 `api/cube/payload.py`를 정리했다.
- 잘못된 payload, 빈 메시지, wake-up 이벤트를 구분해서 처리하도록 `api/cube/service.py`에 검증 로직을 두었다.
- `message_id="-1"` 이고 메시지가 `!@#`로 시작하는 Cube wake-up 이벤트는 실제 대화로 저장하지 않도록 분리했다.

- tests/test_cube_router.py
- tests/test_cube_service.py

3. 동기 처리에서 비동기 큐 구조로 바꾼 일

- 가장 큰 변경은 Cube receiver가 LLM 응답이 끝날 때까지 기다리지 않도록 만든 것이다.
- 이제 receiver는 payload를 검증한 뒤 Redis 큐에 적재하고 바로 `200 OK`를 반환한다.
- 실제 응답 생성과 Cube 재전송은 별도 worker가 수행한다.
- 이 구조로 Cube 플랫폼 쪽 대기 안내 문구 반복, 재시도에 따른 중복 처리 가능성, 긴 요청 시간 문제를 줄이도록 했다.

4. Redis 큐에서 한 일

- ready queue와 processing queue를 분리해 worker가 메시지를 꺼낸 뒤 처리 중 상태를 구분할 수 있게 했다.
- Lua script를 사용해 dedupe key 등록과 enqueue를 원자적으로 처리한다.
- `user_id + message_id` 기준 dedupe를 적용해 같은 메시지가 retry로 다시 들어와도 중복 처리 가능성을 낮췄다.
- worker가 죽거나 재시작된 경우 processing queue에 남아 있던 메시지를 ready queue로 복구하는 로직을 넣었다.
- 실패 시 attempt 값을 증가시켜 재큐잉하고, 최대 retry를 넘기면 error log를 남기고 drop 하도록 만들었다.

관련 코드:

- api/cube/models.py
- api/cube/worker.py
- tests/test_cube_worker.py

5. 실제 응답 처리에서 한 일

- 대화 이력은 `api/conversation_service.py`를 통해 조회하고 사용자 메시지와 assistant 메시지를 순서대로 저장한다.
- 필요하면 선행 안내 문구(`LLM_THINKING_MESSAGE`)를 Cube에 먼저 보내고, 그 다음 실제 workflow 답변을 생성한다.
- 실제 답변 생성은 Cube 계층이 직접 하지 않고 `api/workflows/orchestrator.py`의 `handle_message()`로 넘긴다.
- 답변이 만들어지면 `api/cube/client.py`의 `send_multimessage()`로 Cube `multiMessage` API를 호출한다.
- 이 구조로 Cube 계층은 수신, 큐, 전달에 집중하고, 업무 판단은 workflow 계층으로 분리했다.

6. 설정과 운영 정리에서 한 일

- Cube 연동 설정을 `api/config.py`에 모아 `CUBE_API_ID`, `CUBE_API_TOKEN`, `CUBE_TIMEOUT_SECONDS`, 큐 이름, retry 횟수 등을 환경 변수 기반으로 관리한다.
- 현재 Cube 큐는 별도 Redis URL을 두지 않고 primary Redis인 `REDIS_URL`을 공유하도록 정리했다.
- uWSGI 운영 설정에서 `attach-daemon = python cube_worker.py`로 Cube worker를 웹앱과 함께 띄우는 구성을 사용한다.

7. 모니터링에서 한 일

- `/monitor`에서 Cube Queue 상태와 Cube Worker Daemon 상태를 점검할 수 있게 했다.
- 큐는 enqueue/dequeue/ack roundtrip 검사로 Redis 상태를 확인한다.
- worker는 activity log의 `cube_worker_started`, `cube_worker_heartbeat` 이벤트를 읽어 running, stale, not running 상태를 판별한다.
- 이로써 "웹은 떠 있는데 Cube 응답 worker만 죽은 상태"를 운영에서 빨리 확인할 수 있게 했다.

8. 현재 Cube 처리 흐름

1) Cube가 `/api/v1/cube/receiver`로 메시지를 보낸다.
2) `api/cube/router.py`가 요청을 받고 `api/cube/service.py`의 `accept_cube_message()`를 호출한다.
3) `api/cube/payload.py`가 사용자, 채널, 메시지, message_id를 추출한다.
4) `api/cube/queue.py`가 Redis 큐에 메시지를 적재하고 receiver는 즉시 200 응답을 돌려준다.
5) `cube_worker.py` 또는 `api/cube/worker.py`가 큐에서 메시지를 꺼낸다.
6) `api/cube/service.py`가 대화 이력을 저장하고 `api/workflows/orchestrator.py`로 답변 생성을 위임한다.
7) 생성된 답변을 `api/cube/client.py`가 Cube `multiMessage` API로 전송한다.
8) 로그와 모니터링은 `api/utils/logger/*`, `api/monitoring_service.py`에서 추적한다.

10. 한 문장 결론

우리가 Cube 플랫폼에서 해 온 핵심 작업은 "Cube 메신저 연동을 안정적인 비동기 큐 기반 구조로 바꾸고, workflow 계층과 분리하고, 운영자가 worker/queue 상태를 확인할 수 있게 만든 것"으로 정리할 수 있다.
