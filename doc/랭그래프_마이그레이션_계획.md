# LangGraph 마이그레이션 계획

## Context

프로젝트에 커스텀 워크플로 오케스트레이터(`orchestrator.py`, `state_service.py`, `conversation_service.py`)가 있으며, 워크플로 상태를 JSON 파일로, 대화 이력을 MongoDB에 수동 관리하고 있다. `requirements.txt`에 `langgraph`, `langchain-openai`, `langgraph-checkpoint-mongodb`가 포함되어 있었지만 실제로는 사용되지 않았다. 목표는 LangGraph의 `StateGraph` + MongoDB 체크포인팅으로 전환하여:

1. 대화 이력이 자동으로 누적 (수동 `append_message` 불필요)
2. 워크플로 상태가 체크포인터로 영속화 (JSON 파일 불필요)
3. 핸드오프 패턴(start_chat -> child -> return)이 LangGraph 서브그래프로 구현
4. `thread_id` = `user_id::channel_id` (channel_id가 빈 문자열이면 `user_id`만 사용)

## Phase 0: 인프라 + LLM 클라이언트 마이그레이션 (워크플로 동작 변경 없음) -- 완료

**목표**: 공유 기반을 구축하고 LLM 호출을 ChatOpenAI로 마이그레이션.

### 0a. 체크포인터 & thread_id

1. **`api/workflows/langgraph_checkpoint.py`** -- 기존 파일 확장
   - `get_checkpointer()` 팩토리: `AFM_MONGO_URI` 설정 시 `MongoDBSaver`, 테스트용 `MemorySaver` 반환
   - `build_thread_id(user_id: str, channel_id: str = "") -> str` 유틸리티

2. **`api/workflows/lg_state.py`** (신규) -- 공유 LangGraph 상태 정의
   - `ChatState(TypedDict)`: `messages: Annotated[list, add_messages]`, `user_id`, `channel_id`, `user_message`, `workflow_id`
   - 워크플로별 상태 확장 (TranslatorState, ChartMakerState, TravelPlannerState, StartChatState)

### 0b. LLM 클라이언트 -> ChatOpenAI

3. **`api/llm/service.py`** -- `httpx.post`를 `ChatOpenAI`로 교체
   - `ChatOpenAI(base_url=config.LLM_BASE_URL, model=config.LLM_MODEL, api_key=config.LLM_API_KEY, timeout=config.LLM_TIMEOUT_SECONDS)`
   - `_build_messages()`가 dict 대신 `list[BaseMessage]` (SystemMessage, HumanMessage, AIMessage) 반환
   - `generate_reply()`가 `llm.invoke(messages)` 호출 -> `response.content` 반환
   - 동일한 공개 인터페이스 유지: `generate_reply(history=..., user_message=..., user_profile_text=...)`

4. **`api/llm/registry.py`** -- 실제 `ChatOpenAI` 팩토리로 교체
   - `get_llm(workflow_id, node_id) -> ChatOpenAI` 설정된 LLM 인스턴스 반환

### 테스트

## Phase 1: translator 워크플로 변환 (MCP 도구 포함 가장 단순한 자식)

**목표**: 하나의 워크플로가 LangGraph에서 실행되고 나머지는 커스텀 오케스트레이터에 유지.

### 전략: 어댑터 레이어
- `api/workflows/translator/lg_graph.py` 생성 -- LangGraph `StateGraph` 버전
- `api/workflows/translator/lg_adapter.py` 생성 -- LangGraph 그래프를 기존 `build_graph()` dict 인터페이스로 래핑
- 환경변수 `USE_LANGGRAPH_TRANSLATOR=true`로 토글

### 노드 시그니처 변경
```python
# Before (커스텀)
def entry_node(state: TranslatorWorkflowState, user_message: str) -> NodeResult

# After (LangGraph)
def entry_node(state: TranslatorState) -> dict | Command
# user_message는 state["user_message"]
# NodeResult 대신 상태 업데이트 dict 반환
```

### 핵심 매핑
| 커스텀 | LangGraph |
|--------|-----------|
| `NodeResult(action="resume", next_node_id="X")` | `Command(goto="X")` 또는 조건부 엣지 |
| `NodeResult(action="wait", reply="...")` | `interrupt({"reply": "..."})` |
| `NodeResult(action="complete", reply="...")` | Return + `END` 엣지 |
| `state.source_text` | `state["source_text"]` |

### 생성/수정 파일
- `api/workflows/translator/lg_graph.py` (신규)
- `api/workflows/translator/lg_adapter.py` (신규)
- `api/workflows/translator/nodes.py` -- 시그니처 적응 (어댑터로 기존 것도 동작 유지)
- `api/workflows/translator/__init__.py` -- 이전/신규 그래프 토글
- LangGraph translator 그래프 테스트

## Phase 3: start_chat + 새 오케스트레이터 변환 (핵심 단계)

**목표**: 서브그래프를 통한 핸드오프 포함, 전체 흐름이 LangGraph에서 실행.

### 아키텍처
```
start_chat StateGraph
  +-- entry_node
  +-- classify_node --> conditional_edge
  |   +-- "start_chat" -> retrieve_context -> plan -> generate -> END
  |   +-- "translator" -> translator_subgraph -> END
  |   +-- "chart_maker" -> chart_maker_subgraph -> END
  |   +-- "travel_planner" -> travel_planner_subgraph -> END
  +-- compiled with checkpointer
```

### 새 오케스트레이터 진입점
- `api/workflows/lg_orchestrator.py` 생성
  - `handle_message(incoming) -> WorkflowReply`
  - `user_id + channel_id`로 `thread_id` 생성
  - `graph.invoke({"messages": [HumanMessage(...)], "user_message": ...}, config)` 호출
  - 마지막 AI 메시지에서 답변 추출
  - `WorkflowReply(reply=..., workflow_id=state["workflow_id"])` 반환

### cube/service.py 업데이트
- `USE_LANGGRAPH=true` 토글로 기존 오케스트레이터에서 `lg_orchestrator.handle_message`로 전환
- LangGraph 활성 시, 사용자/어시스턴트의 수동 `append_message` 생략 -- 체크포인터가 처리
- `get_history()` 호출은 로깅용 `history_length`만 위해 유지 (또는 제거)

### 핵심 결정
- **conversation_service.py 유지** -- `get_recent_messages()` (모니터링 페이지)와 감사 기록으로 유용
- **state_service.py 우회** -- 체크포인터가 JSON 파일 영속화 대체
- **핸드오프 스택 제거** -- 서브그래프가 자연스럽게 진입/퇴출 처리

### 생성/수정 파일
- `api/workflows/start_chat/lg_graph.py` (신규)
- `api/workflows/lg_orchestrator.py` (신규)
- `api/cube/service.py` -- 오케스트레이터 토글

- 전체 LangGraph 흐름 테스트

## conversation_service.py -- 감사 기록으로 유지

- 체크포인터가 LLM 히스토리를 자동 처리 (누적 + 조회)
- `conversation_service.py`는 별도로 MongoDB에 메시지 저장 계속:
  - 모니터링 페이지 (`get_recent_messages()`)
  - 감사/분석 쿼리 (예: `workflow_id`, `direction`, `channel_id` 필터)
- `cube/service.py`가 감사용 `append_message()` 계속 호출, LLM 히스토리는 체크포인터에서
- 대화 컬렉션의 TTL 인덱스 유지

## TTL / 정리 고려사항

LangGraph 체크포인터에는 내장 TTL이 없다. 체크포인트 컬렉션에 MongoDB TTL 인덱스 추가:
```python
db["checkpoints"].create_index("ts", expireAfterSeconds=CONVERSATION_TTL_SECONDS)
```

## Thread ID 전략

```python
def build_thread_id(user_id: str, channel_id: str = "") -> str:
    if channel_id:
        return f"{user_id}::{channel_id}"
    return user_id
```
