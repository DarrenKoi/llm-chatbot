브랜치: main, 마지막 커밋 `019aab8` (이번 세션에서 작성)

## 목표 (사용자 요청)

Cube 장애 대비/브라우저 선호 사용자용 웹 채팅 채널이 실제로 동작하는지 확인.
기능 자체는 이미 `f542cea`에서 전부 구현·커밋되어 있었음 (backend `api/web_chat/`,
SPA 소스 `web/`, 정적 서빙 `api/web_frontend/`). 사용자가 선택한 작업 모드:
**"Build, deploy & verify"** — 새 기능 추가 아님.

## 완료된 것

1. **Nuxt 빌드 & 배포 절차 확인됨**: `cd web && npm run build` →
   `web/.output/public/` 전체를 `api/static/chat/`로 복사. (산출물은 gitignore 대상)
2. **치명적 프론트 버그 발견·수정·커밋 (`019aab8`)**:
   `components/chat/ConversationHeader.vue`/`ConversationList.vue`가 Nuxt 경로
   prefix 규칙 때문에 `ChatConversationHeader`/`ChatConversationList`로 등록되어,
   `pages/index.vue`의 `<ConversationHeader>`/`<ConversationList>`가 프로덕션
   빌드에서 **조용히** 미해석 커스텀 엘리먼트로 렌더링됨 (헤더·대화목록 사라짐).
   `ChatTimeline`/`ChatComposer`는 파일명이 이미 Chat으로 시작해 prefix가 dedupe
   되어 정상 동작 → 절반만 깨지는 형태라 리뷰에서 안 보였던 것.
   수정: `web/nuxt.config.ts`에 `components: [{ path: '~/components', pathPrefix: false }]`.
   **수정 후 재빌드·재배포까지 완료, 브라우저에서 전체 UI 정상 렌더링 확인됨.**
3. **Playwright로 검증 완료된 항목**:
   - `/chat` 페이지 로드, SPA 부팅, CSS/자산 로딩 (`/chat/_nuxt/*` 경로 정상)
   - 헤더(사용자명, 연결됨 배지), 사이드바, "새 대화" 버튼, 타임라인, 컴포저 렌더링
   - 사용자 메시지 전송 → user 버블 표시 → "응답 생성 중..." 표시
   - 대화 목록에 대화 + 미리보기 노출, 재접속 시 이력 로드 (서버 재시작에도
     대화가 살아 있었음 — REDIS_URL 빈 값 fallback이 순수 in-memory가 아닐 가능성,
     확인할 가치 있음)
   - 워크플로 실패 시 assistant 메시지 미저장 (설계 의도대로) 확인
4. **API 레벨 검증 완료**:
   - `GET /me`: LASTUSER 쿠키 → 200, 쿠키 없음+localhost+`WEB_CHAT_DEV_USER` → dev user,
     쿠키 없음+비-localhost Host → 401 (3 경로 모두 정상)
   - `POST .../messages` 빈/비JSON body → 400, LLM 불통 시 → 502 `workflow_failed`

## 막힌 지점 (다음 세션이 이어서 할 일)

**유일하게 미검증**: assistant 응답이 타임라인에 실제 렌더링되는 것.

집에는 LLM이 없어서 mock LLM 서버를 만들었음:
`C:\Users\Daeyoung\AppData\Local\Temp\mock_llm_server.py` (127.0.0.1:8765,
OpenAI 호환 `/v1/chat/completions`, tool-call 미지원이라 `generate_reply_intent`의
plain-text fallback 경로를 태움 — TextIntent로 감싸져 응답 보장됨).

마지막 실패 원인(추정 높음): mock이 단일스레드 `HTTPServer` + HTTP/1.0이라
keep-alive를 안 지켜서, httpx 커넥션 풀이 재사용하는 두 번째 요청(structured
output 실패 후 plain invoke fallback)이 connection error. **이미 수정해 둠**
(`ThreadingHTTPServer` + `protocol_version = "HTTP/1.1"`) — 수정 후 아직 미실행.

### 재개 절차 (그대로 실행하면 됨)

```bash
# 1. mock LLM (반드시 sandbox 비활성화로 실행 — 아래 함정 참조)
python C:\Users\Daeyoung\AppData\Local\Temp\mock_llm_server.py

# 2. Flask (역시 sandbox 비활성화)
cd C:/Code/llm-chatbot
LLM_BASE_URL=http://127.0.0.1:8765/v1 LLM_MODEL=mock-model LLM_API_KEY=test-key \
  REDIS_URL= AFM_MONGO_URI= python index.py

# 3. 확인: 메시지 전송 후 assistant 버블 렌더링
curl -s -X POST http://127.0.0.1:5000/api/v1/web-chat/conversations/t1/messages \
  -H "Content-Type: application/json" -b "LASTUSER=tester" \
  -d "{\"message\":\"hello\"}"
# 기대: {"reply":"[mock-llm] ...","conversation_id":...} 200
# 그 다음 Playwright로 http://127.0.0.1:5000/chat 에서 전송→응답 버블 확인
```

### 이번 세션에서 배운 함정 (시간 잡아먹은 것들)

1. **Claude Code sandbox가 loopback을 격리함**: sandbox 안에서 띄운 서버의
   포트는 sandbox 밖 프로세스에서 connection refused. Bash로 서버 띄울 때
   Flask·mock 둘 다 `dangerouslyDisableSandbox: true` 필요. (sandbox 안
   curl/httpx로는 성공해서 진단이 헷갈렸음)
2. **werkzeug debug 모드는 프로세스 2개** (reloader 부모 + 서버 자식).
   하나만 죽이면 자식이 계속 포트를 서빙하고, Windows에선 새 프로세스가
   같은 포트에 또 bind돼 버려서 "고쳤는데 그대로"처럼 보임. 종료 시
   `Get-CimInstance Win32_Process`로 `index.py` 프로세스 전부 확인 후 kill.
3. git-bash에서 curl로 한글 JSON 보내면 인코딩 깨져 400 → 테스트는 ASCII로.
4. 이 PC에 `WEB_CHAT_DEV_USER=devuser`가 시스템 환경변수로 설정돼 있음
   (쿠키 없이도 localhost에서 devuser로 동작). 운영 검증 시 주의.
5. Flask 로그에 `profile_redis_lookup_failed ... Timeout connecting to server`
   — `REDIS_URL=`를 비웠는데도 profile 서비스가 어딘가로 Redis 연결을 시도해
   ~5초 지연 발생. 메시지 전송 경로의 체감 지연 원인. 별도 확인 가치 있음.
