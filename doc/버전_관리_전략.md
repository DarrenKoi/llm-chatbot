# 버전 관리 전략

## 현재 상태

- **앱 버전 관리 없음** — VERSION 파일, `__version__`, git tag 없음
- **라우트 버전은 있음** — `/api/v1/cube/receiver`, `/api/v1/file-delivery/upload` 등
- **라우트 버전이 데코레이터에 하드코딩** — Blueprint `url_prefix` 미사용
- **배포 롤백 불가** — `/project/workSpace/`에 직접 배포, symlink 구조 없음

---

## 무엇을 버전 관리해야 하는가?

### API 라우트만 버전 관리하면 되는가? — 아니다

라우트만 버전 관리하면 이런 문제가 생긴다:

```
api/cube/router.py      → /api/v1/cube/receiver  (v1 라우트)
api/cube/router_v2.py   → /api/v2/cube/receiver  (v2 라우트)
api/cube/service.py     → 두 라우터 모두 이 서비스를 호출
```

`service.py`를 v2에 맞게 변경하면 **v1이 깨진다**. 라우트 버전은 HTTP 인터페이스만 보호하고, 그 뒤의 로직은 보호하지 않는다.

### 현실적인 3단계 접근

변경의 규모에 따라 다른 전략이 필요하다:

| 변경 규모 | 버전 관리 범위 | 예시 |
|-----------|--------------|------|
| **소규모** — 파라미터 추가, 응답 필드 추가 | 라우트만 (기존 서비스에 기본값 파라미터 추가) | 새 필드 `priority`를 응답에 추가 |
| **중규모** — 서비스 로직 변경 | 라우트 + 서비스 함수 분리 | 메시지 처리 파이프라인 변경 |
| **대규모** — 패키지 전체 재설계 | 패키지 복제 (`api/v2/cube/`) | Cube 연동 아키텍처 전면 교체 |

---

## 전략 1: 라우트 버전 관리 (소규모 변경)

서비스 계층이 하위 호환되는 경우. 새 파라미터에 기본값을 주면 v1 라우터가 깨지지 않는다.

### 현재 구조 리팩토링

라우트 데코레이터에서 `/api/v1`을 제거하고 Blueprint `url_prefix`로 이동:

```python
# api/cube/router.py — 변경 전
bp = Blueprint("cube", __name__)

@bp.route("/api/v1/cube/receiver", methods=["POST"])
def receive_cube(): ...

# api/cube/router.py — 변경 후
bp = Blueprint("cube", __name__, url_prefix="/api/v1/cube")

@bp.route("/receiver", methods=["POST"])
def receive_cube(): ...
```

### v2 라우터 추가 (필요할 때)

```python
# api/cube/router_v2.py — 새 파일
bp = Blueprint("cube_v2", __name__, url_prefix="/api/v2/cube")

@bp.route("/receiver", methods=["POST"])
def receive_cube_v2():
    # 같은 service.py를 호출하되, 새 파라미터 전달
    result = accept_cube_message(request.get_json(), version=2)
    return jsonify(result.to_v2_dict()), 200
```

`blueprint_loader.py`가 `router_*.py`를 자동 탐색하므로 별도 등록 불필요.

**적합한 경우:** 응답 형식 변경, 새 필드 추가, 선택적 파라미터 추가
**부적합한 경우:** 내부 모델이나 처리 흐름 자체가 바뀔 때

## 전략 2: 서비스 계층 분리 (중규모 변경)

서비스 로직이 달라져야 할 때. 서비스 함수를 버전별로 분리한다.

### 방법 A: 함수 분기

```python
# api/cube/service.py
def accept_cube_message(payload, *, version=1):
    incoming = _parse_incoming(payload)
    if version == 1:
        return _handle_v1(incoming)
    return _handle_v2(incoming)
```

단순하지만 버전이 3개 이상이면 지저분해진다.

### 방법 B: 서비스 파일 분리 (권장)

```
api/cube/
  router.py          # v1 라우트
  router_v2.py       # v2 라우트
  service.py         # 공유 유틸 + v1 핸들러
  service_v2.py      # v2 전용 핸들러 (공유 유틸 import)
  models.py          # 공유 모델
  client.py          # 공유 Cube API 클라이언트
```

```python
# api/cube/service_v2.py
from api.cube.service import _parse_incoming, _send_reply  # 공유 부분 재사용
from api.cube.models import CubeIncomingMessage

def accept_cube_message_v2(payload):
    incoming = _parse_incoming(payload)
    # v2 전용 로직
    ...
```

**핵심 원칙:** 공유 가능한 것(클라이언트, 모델, 유틸)은 공유하고, 분기가 필요한 것(서비스 핸들러)만 분리한다.

**적합한 경우:** 처리 파이프라인 변경, 워크플로우 로직 변경
**부적합한 경우:** 모델, 클라이언트, 큐 구조까지 전부 바뀔 때

---

## 전략 3: 패키지 복제 (대규모 변경)

전면 재설계가 필요할 때. v1 패키지를 통째로 유지하면서 v2를 새로 만든다.

```
api/
  cube/              # v1 — 건드리지 않음 (동결)
    router.py
    service.py
    models.py
    ...
  cube_v2/           # v2 — 완전히 새로운 구현
    router.py
    service.py
    models.py
    ...
```

**장점:** v1이 절대 깨지지 않음 (코드 자체가 분리)
**단점:** 코드 중복, 버그 수정을 양쪽에 해야 함
**적합한 경우:** 아키텍처 전면 교체, v1 단계적 폐기 예정

## 이 프로젝트에 권장하는 접근

현재 프로젝트 규모와 팀 크기(1인)를 고려하면:

1. **지금 당장:** 전략 1 (라우트 url_prefix 리팩토링) — 이것만으로 향후 v2 추가 준비 완료
2. **서비스 로직이 바뀔 때:** 전략 2B (service_v2.py 분리) — 공유 부분 재사용
3. **전면 재설계 시:** 전략 3 — 하지만 가능하면 피할 것

**실질적으로 가장 자주 쓸 패턴은 전략 2B**이다. 라우트만 바뀌는 경우는 드물고, 보통 서비스 로직도 함께 바뀌기 때문이다.

## 앱 버전 관리

API 버전(v1, v2)과 별개로, **배포된 코드가 어떤 버전인지** 추적해야 한다.

### VERSION 파일

프로젝트 루트에 `VERSION` 파일 생성:

```
1.0.0
```

SemVer 규칙:
- **MAJOR** — Cube 웹훅 계약 변경 (외부 호환성 깨짐)
- **MINOR** — 새 기능, 새 워크플로우 추가
- **PATCH** — 버그 수정

### 런타임 노출

```python
# api/config.py
APP_VERSION = (BASE_DIR / "VERSION").read_text().strip()
```

```python
# api/__init__.py — _after_request 훅에 추가
response.headers["X-App-Version"] = config.APP_VERSION
```

`/health` 엔드포인트 추가:
```python
@app.route("/health", methods=["GET"])
def health():
    return jsonify({"status": "ok", "version": config.APP_VERSION})
```

### Git 태그

버전 범프 시 태그 생성:
```bash
echo "1.2.0" > VERSION
git add VERSION
git commit -m "release: v1.2.0 배포 준비"
git tag v1.2.0
git push origin main --tags
```

## 배포 버전 관리

### Symlink 기반 릴리스

```
/project/workSpace/
  releases/
    v1.0.0/
    v1.1.0/
  current -> releases/v1.1.0   # symlink
  shared/
    .env
    logs/
    var/                        # workflow_state, conversation_history, mcp_cache
    file_delivery/
```

### 배포 스크립트 (`scripts/deploy.sh`)

```bash
#!/bin/bash
set -euo pipefail
VERSION=${1:?Usage: deploy.sh <version-tag>}
DEPLOY_ROOT=/project/workSpace
RELEASE_DIR=$DEPLOY_ROOT/releases/$VERSION

git clone --branch "$VERSION" --depth 1 <repo-url> "$RELEASE_DIR"

# 공유 리소스 심링크
ln -sfn $DEPLOY_ROOT/shared/.env "$RELEASE_DIR/.env"
ln -sfn $DEPLOY_ROOT/shared/logs "$RELEASE_DIR/logs"
ln -sfn $DEPLOY_ROOT/shared/var "$RELEASE_DIR/var"

pip install -r "$RELEASE_DIR/requirements.txt"

# 원자적 심링크 교체
ln -sfn "$RELEASE_DIR" "$DEPLOY_ROOT/current"
touch "$DEPLOY_ROOT/current/restart.txt"

echo "Deployed $VERSION"
```

### 롤백

```bash
ln -sfn /project/workSpace/releases/v1.0.0 /project/workSpace/current
touch /project/workSpace/current/restart.txt
```

## 요약

- **라우트 버전만으로는 부족하다** — 서비스 로직 변경 시 v1이 깨질 수 있다
- **서비스 파일 분리(service_v2.py)가 가장 현실적인 패턴** — 공유 부분 재사용, 분기 부분만 분리
- **패키지 복제는 최후 수단** — 코드 중복 비용이 크다
- **앱 버전(VERSION + git tag)과 API 버전(v1/v2)은 별개** — 둘 다 필요하다
