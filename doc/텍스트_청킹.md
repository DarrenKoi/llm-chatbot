# 텍스트 청킹 & 전송 라우팅

## 배경

Cube 플랫폼의 `multiMessage` API는 약 45줄에서 강제 줄바꿈을 삽입한다.
LLM이 생성한 긴 응답이 의도치 않게 잘리는 문제가 발생한다.

반면 `richnotification` API는 줄 수 제한이 없거나 훨씬 넉넉하다.

## 해결 전략

LLM 응답을 콘텐츠 유형별로 분리하여 적절한 API로 전송한다.

| 콘텐츠 유형 | 전송 API | 청킹 |
|---|---|---|
| 일반 텍스트 | `multiMessage` | 40줄 단위 청킹 |
| 코드 블록 (` ``` `) | `richnotification` | 불필요 |
| 표 (`\|---\|` 패턴) | `richnotification` | 불필요 |

## 처리 흐름

```
LLM 응답 → 블록 파싱 → 전송 방법 라우팅 → 텍스트 블록 청킹 → 순서대로 전송
```

1. **블록 파싱**: 응답 텍스트를 코드 펜스, 표, 일반 텍스트 블록으로 분리
2. **라우팅**: 코드/표 → richnotification, 텍스트 → multimessage
3. **청킹**: multimessage 블록만 40줄 한도 내에서 문단/헤더 경계로 분할
4. **전송**: `DeliveryItem` 리스트를 순서대로 Cube API 호출

## 설정

| 환경변수 | 기본값 | 설명 |
|---|---|---|
| `CUBE_MESSAGE_MAX_LINES` | `40` | multimessage 청크당 최대 줄 수 |
| `CUBE_RICH_ROUTING_ENABLED` | `false` | `true`로 설정 시 코드/표를 richnotification으로 전송 |

> **현재 상태:** rich routing은 OFF (기본값). richnotification 전송 구현이 완료되면 `CUBE_RICH_ROUTING_ENABLED=true`로 활성화.

## 주요 파일

- `api/cube/chunker.py` — 청킹 + 라우팅 로직
- `api/cube/service.py` — 전송 통합 (line ~292)
- `api/config.py` — 설정
- `tests/test_cube_chunker.py` — 테스트

## 텍스트 품질 보장

- 코드 블록은 절대 중간에서 자르지 않음 (richnotification으로 통째로 전송)
- 표도 마찬가지로 원자 단위 보존
- 일반 텍스트 청킹 시 문단 경계(`\n\n`), 마크다운 헤더(`## `) 우선 분할
- 줄 경계 분할은 최후 수단
