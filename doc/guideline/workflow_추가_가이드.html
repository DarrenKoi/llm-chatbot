1. 실행 흐름 한눈에 보기

새 workflow는 공통 엔진을 바꾸는 작업이 아니라, 기존 파이프라인 끝에 새 패키지를 붙이는 작업이다.

Input

Cube Message

사용자 메시지가 `api/cube/router.py`를 통해 큐에 들어간다.

Infra

Queue / Worker

worker가 메시지를 꺼내고 오케스트레이터에 넘긴다.

Core

Orchestrator

`run_graph()`가 현재 `node_id`를 기준으로 node를 실행한다.

Feature

Workflow Package

`api/workflows/<workflow_id>/`의 node들이 실제 업무 로직을 처리한다.

Contract

NodeResult

`resume`, `wait`, `handoff`, `complete`에 따라 다음 상태와 응답이 결정된다.

2. 무엇을 만들고 무엇은 건드리지 않는가

동료가 가장 자주 헷갈리는 지점이다. 기본 원칙은 새 패키지를 추가하고, 공통 엔진 파일은 웬만하면 건드리지 않는 것이다.

보통 여기만 추가하면 된다

  • `api/workflows/travel_planner/` 같은 새 패키지
  • workflow 전용 상태 dataclass
  • node 함수와 graph 정의
  • workflow 테스트 파일
  • 필요하면 `handoff_keywords`

추가 작업만 할 때는 피하는 편이 안전하다

  • `api/workflows/registry.py` 수동 수정
  • `api/workflows/orchestrator.py` 구조 변경
  • `api/workflows/start_chat/` 분류 규칙 변경
  • 기존 workflow 패키지 내부 직접 수정
  • 공통 상태 저장 규약 변경

3. 최소 패키지 구조

새 workflow를 만들 때는 파일 수를 늘리기보다 역할을 분명히 나누는 것이 더 중요하다.

__init__.py

등록 정의

`get_workflow_definition()`에서 `workflow_id`, `entry_node_id`, `build_graph`, `state_cls`, `handoff_keywords`를 노출한다.

state.py

상태 필드

`WorkflowState`를 상속해 턴 간에 유지할 값만 단순 자료형으로 정의한다.

nodes.py

실제 대화 로직

입력 해석, 재질문, 외부 연동, 완료 응답을 node 함수로 나눈다.

graph.py

흐름 정의

entry node와 node 맵, 시각화용 edge를 묶어 반환한다.

4. 실전 샘플: travel_planner

`travel_planner`는 동료가 가장 쉽게 베껴서 새 workflow를 만들 수 있도록 의도적으로 단순한 규칙 기반 예제로 구현했다.

api/workflows/travel_planner/

샘플의 핵심 역할

이 샘플은 목적지 추출, 취향 재질문, 추천 후보 제안, 최종 장소 추천까지 한 사이클을 모두 보여준다. 즉, 새 workflow에서 자주 필요한 상태 업데이트와 multi-turn 흐름을 한 번에 확인할 수 있다.

State fields

`destination`, `travel_style`, `duration_text`, `companion_type`, `suggested_destinations`

Actions used

`wait`로 재질문하고, `resume`으로 다음 node를 이어가며, 마지막에 `complete`로 종료한다.

Handoff

`여행 계획`, `trip plan`, `travel planner` 같은 키워드로 `start_chat`에서 진입 가능하다.

tests/test_travel_planner_workflow.py

동료가 바로 복제할 수 있는 테스트 패턴

테스트는 "첫 질문", "후보 추천", "충분한 정보가 들어왔을 때 완료", "multi-turn 이어달리기" 네 가지를 고정한다. 새 workflow를 만들 때도 이 네 칸을 먼저 채우면 흐름이 흔들리지 않는다.

Step 1

사용자 요청이 너무 넓으면 취향부터 묻는다

`entry` node가 목적지와 일정이 비어 있음을 확인하면 `wait`를 반환하고 여행 스타일을 질문한다.

Step 2

스타일만 있으면 목적지 후보를 추천한다

`recommend_destination` node가 "도시 / 휴양 / 자연 / 먹거리"에 따라 3개 후보를 제안한다.

Step 3

목적지가 정해지면 일정 정보를 받는다

목적지는 있는데 기간이 없으면 `collect_trip_context`에서 `2박 3일` 같은 입력을 다시 받는다.

Step 4

방문 장소를 추천하고 종료한다

`build_plan` node가 목적지별 대표 장소 3곳과 간단한 동선 팁을 묶어 `complete`한다.

6. NodeResult를 어떻게 읽으면 되는가

새 workflow가 어렵게 느껴지는 이유는 결국 이 계약이 눈에 잘 안 들어오기 때문이다. 아래 세 가지를 먼저 익히면 된다.

`wait`

사용자에게 질문을 던지고 멈춘다. `reply`와 `next_node_id`를 같이 넣는 것이 안전하다.

`resume`

같은 턴 안에서 다음 node를 바로 실행한다. 정보가 충분할 때 내부적으로 흐름을 넘길 때 쓴다.

`complete`

현재 workflow를 끝낸다. handoff된 자식 workflow라면 부모 workflow 복귀 조건이 되기도 한다.

7. 동료용 실무 체크리스트

이 다섯 줄만 맞추면 새 workflow PR의 대부분이 정리된다.

  • 패키지명과 `workflow_id`가 일치하는지 확인한다.
  • 상태 필드는 JSON 직렬화 가능한 기본형만 둔다.
  • 정보가 부족할 때는 `wait + reply + next_node_id`를 같이 준다.
  • `handoff_keywords`는 넓지 않게, 실제 사용자 표현 위주로 넣는다.
  • 최소한 기본 흐름과 multi-turn 테스트는 함께 추가한다.
