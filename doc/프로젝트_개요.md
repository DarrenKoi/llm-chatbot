# LLM Chatbot 프로젝트 개요

## 한 줄 요약

사내 메신저 Cube에 연동되는 **LangGraph 기반 AI 챗봇 서버**로, 사내 인프라 안에서 대화 처리, MCP 도구 호출, RAG 검색을 함께 담당합니다.

---

## 배경 및 목적

### 문제 인식

- 업무 중 반복적인 데이터 조회, 시각화 요청이 발생하지만, 이를 위해 별도 시스템에 접속하거나 담당자에게 요청해야 하는 비효율 존재
- 외부 AI 서비스(OpenAI, Anthropic 등)는 보안 정책상 사내 데이터를 전송할 수 없음

### 해결 방향

- **사내 로컬 LLM**을 활용하여 보안 이슈 없이 AI 기능을 제공
- 이미 사용 중인 **Cube 메신저**에 통합하여 별도 앱 설치 없이 자연어로 즉시 이용 가능
- **LangGraph** 기반 workflow로 conditional routing, MCP tool calling, RAG 검색을 유연하게 오케스트레이션
- 대화 이력은 **LangGraph checkpointer**가 자동 관리하고, 완료된 대화는 **OpenSearch**에 아카이빙

---

## 주요 기능

| 기능 | 설명 |
|------|------|
| **자연어 대화** | 한국어 질문에 대해 LLM이 답변 생성 |
| **대화 기록 유지** | LangGraph checkpointer(MongoDB)가 대화 맥락을 자동 저장/복원 |
| **다중 모델 지원** | 사내 LLM 모델(Kimi-K2.5, Qwen3, GPT-OSS 등)을 task/조건에 따라 선택 |
| **MCP 도구 호출** | MCP 서버를 통해 외부 도구를 호출하여 실시간 데이터 조회, 업무 시스템 연동 |
| **대화 아카이빙** | 완료된 대화에서 키워드/만족도 추출 후 OpenSearch에 인덱싱 |
| **사용자 명령어** | `!model`, `!remove` 등 `!` 접두사 명령으로 모델 변경, 히스토리 삭제 |
| **비동기 메시지 처리** | Cube 수신 요청은 큐에 적재하고 worker가 백그라운드에서 처리 |
| **Cube 메신저 연동** | Cube에서 메시지를 받아 처리 후 `multiMessage`로 응답 |
| **CDN 파일 처리** | 파일 업로드, 원본 다운로드, 이미지 리사이즈/썸네일 제공 |
| **운영 로그/스케줄러** | 구조화 activity 로그와 APScheduler 기반 백그라운드 작업 지원 |

## 기술 스택

| 구분 | 기술 |
|------|------|
| **백엔드** | Python, Flask, uWSGI |
| **LLM** | OpenAI 호환 로컬 LLM (langchain-openai `ChatOpenAI`) |
| **워크플로** | LangGraph (conditional routing, tool loop) |
| **도구 호출** | MCP (Model Context Protocol) |
| **대화 저장** | LangGraph checkpointer (langgraph-checkpoint-mongodb) |
| **대화 아카이빙** | OpenSearch (키워드, 만족도, 검색) |
| **메시지 큐** | Redis |
| **데이터 저장** | MongoDB |
| **로그 저장** | 파일 기반 JSONL + stdout |
| **메신저 연동** | Cube API |
| **스케줄러** | APScheduler |

## 시스템 구성

```
사용자
  │
  ▼
Cube 메신저
  │ POST /api/v1/cube/receiver
  ▼
LLM Chatbot 서버 (Flask + uWSGI)
  ├─ cube/            HTTP 수신, 큐 적재, 응답 전송
  ├─ workflows/       LangGraph StateGraph 기반 오케스트레이션
  │   ├─ start_chat/    루트 워크플로 (분류 → 일반 대화 or 서브그래프 handoff)
  │   ├─ translator/    번역 워크플로 (MCP 도구 호출)
  │   ├─ chart_maker/   차트 생성 워크플로
  │   └─ travel_planner/ 여행 계획 워크플로
  ├─ llm/             모델 호출 (langchain-openai ChatOpenAI)
  ├─ mcp/             MCP 도구 인프라 (로컬 핸들러 + 원격 클라이언트)
  ├─ profile/         사용자 프로필 로드 (API → Redis fallback)
  ├─ archive/         대화 아카이빙 (OpenSearch)
  ├─ file_delivery/   파일 업로드/다운로드
  ├─ scheduled_tasks/ APScheduler 기반 정기 작업
  └─ utils/           로그
      │
      ├─ Redis          Cube 수신 큐, 파일 메타데이터, 스케줄러 락
      ├─ MongoDB        LangGraph 체크포인터, 대화 이력 감사 기록
      ├─ OpenSearch     대화 아카이브, 검색/분석
      ├─ MCP 서버       외부 도구 서비스 (별도 서비스)
      └─ 로컬 LLM       OpenAI 호환 LLM 서버
```

## 핵심 설계 원칙

1. **외부 AI 서비스 의존 제로** — 모든 LLM 호출은 사내 로컬 서버로만 전송
2. **업무 workflow 단위 설계** — 각 업무(번역, 차트, 여행 등)가 독립 패키지로 존재하고, `start_chat`이 루트 그래프로서 분류 + handoff 담당
3. **비동기 응답 처리** — Cube 요청은 큐에 적재하고 worker가 백그라운드에서 처리
4. **환경 변수 기반 설정** — 모든 URL, 인증 정보, 모델명은 환경 변수로 관리
5. **대화 히스토리 자동화** — LangGraph checkpointer(MongoDB)가 저장/복원을 자동 처리, 수동 관리 불필요
6. **MCP 확장형 구조** — MCP 서버가 늘어나도 독립 인프라 계층에서 관리
7. **multi-turn interrupt/resume** — LangGraph `interrupt()` 기반으로 부족한 정보를 재질문하고, 다음 턴에서 `Command(resume=...)` 로 재개

## 기대 효과

- **업무 효율화** — Cube 안에서 자연어로 빠르게 질의응답 가능
- **보안 준수** — 사내 데이터가 외부로 유출되지 않음
- **접근성 향상** — 별도 도구 없이 기존 Cube 메신저에서 바로 사용
- **도구 연동** — MCP를 통해 업무 시스템과 직접 연결하여 실시간 데이터 활용
- **품질 개선** — 대화 아카이브 분석으로 지속적 응답 품질 향상
- **확장 가능** — RAG subgraph, Agent workflow 등 점진적 기능 확장
